4.6 Sticky Sessions — Trade-offs and Alternatives

00 🗺️ Topic Overview

What This Topic Is About

Sticky sessions (session affinity) is a load balancer configuration that routes a client consistently to the same backend server across multiple requests. It exists because some services store session state locally on the server — without affinity, a request landing on a different server finds no session and fails. The central tension is that stickiness restores stateful behavior on top of a load-balanced tier, directly undermining the horizontal scalability that load balancing is supposed to enable.

🎯 What to Focus On

  • 1. Why stickiness is needed in the first placeIt only exists as a band-aid over server-side session state. Understand the root cause (stateful server design), not just the symptom. The correct fix is stateless redesign, not better stickiness.
  • 2. Cookie-based vs. IP-hash affinity mechanismsCookie-based (LB inserts Set-Cookie header) works at L7 and survives IP changes. IP-hash (hash of source IP % num_backends) works at L4/L7 but collapses catastrophically under NAT. Know the failure modes of each.
  • 3. The scaling trapAdding backend nodes doesn't help existing sticky users. Node failure drops all pinned sessions with no recovery. Load skews toward "hot" nodes over time. Interviewers specifically probe this.
  • 4. The three correct alternativesRedis session store (externalized state), JWT stateless auth (client carries state), and stateless service redesign (no server-side state at all). Be able to propose and justify each in an interview.
  • 5. When stickiness is pragmatically acceptableLegacy migration, COTS software, and WebSocket transport-layer affinity (protocol requirement, not a design smell). Know the distinction between unavoidable and avoidable stickiness.

05 🧠 Core Definition

What is it, in one sentence?

Sticky sessions (session affinity) is a load balancer configuration that consistently routes all requests from a given client to the same backend server, allowing that server to serve the client's locally-stored session state across multiple requests — at the cost of free horizontal scaling and per-session fault tolerance.

06 📦 Core Concepts

The essential building blocks — terms and ideas you must have solid before going deeper.

Server-Side Session State

When a server stores user session data (auth tokens, shopping cart, multi-step workflow state) in its own process memory or local disk, that data is only accessible on that specific server. If the next request lands on a different backend, the session is lost — this is the root problem sticky sessions paper over. Without affinity, every stateful server-side request requires either state replication across nodes or centralized state storage.

Cookie-Based Affinity

The load balancer injects a special affinity cookie (e.g., AWSALB for AWS ALB, SERVERID for HAProxy) into the HTTP response. On every subsequent request, the client sends this cookie back and the LB routes directly to the encoded backend — bypassing the normal routing algorithm. This is L7-only (the LB must be HTTP-aware). It survives client IP changes (mobile network switches, VPN connects) making it more robust than IP-hash for most use cases.

IP-Hash Affinity

The LB computes hash(source_IP) % num_backends and routes deterministically — no cookie required, works at L4 or L7. Critically, this computation changes when backends are added or removed, causing mass session disruption on every scaling event. It also degrades severely under corporate NAT or CGNAT (carrier-grade NAT), where thousands of users share one public IP and all route to a single backend.

Session Draining (Connection Draining)

When a backend must be removed (deployment, failure recovery), active sticky sessions cannot be instantly cut off without data loss. Draining stops sending new sessions to the departing node while allowing in-flight sticky sessions to complete. The LB monitors active connections and removes the node once they reach zero (or a configurable timeout). This helps for planned maintenance but offers no protection during sudden node failures.

Stateless Alternatives

The architecturally correct remedy for sticky sessions. Options: (1) Redis session store — externalize session objects so any backend can serve any request; (2) JWT auth — client carries all session state in a cryptographically signed token, no server-side storage; (3) stateless service redesign — eliminate all server-side session state so affinity is never needed. All three restore true horizontal scalability and per-node redundancy.

07 🔍 First Principles — Why Does This Exist?

What fundamental problem does this concept solve? Why was it invented?

HTTP is stateless by design — each request carries no memory of prior requests. Web applications needed persistent sessions (login state, cart contents, multi-step workflows) so they began storing session objects in server memory, keyed by a session ID cookie. This worked fine with a single server.

When load balancers appeared to distribute traffic across a pool of servers, the problem surfaced immediately: if server A issued session ID abc123 and stored it in memory, a subsequent request carrying abc123 landing on server B would find no matching session and fail with a 401 or empty cart.

The immediate fix: tell the LB "once you've sent a client to server A, keep sending them there." This preserved existing application code with zero refactoring. Sticky sessions were invented as a compatibility shim between stateful application architectures and the load-balanced multi-server deployments that scale required. The cost was that horizontal scalability benefit was partially surrendered — you added more servers, but existing users didn't benefit from them.

08 🗺️ Mental Models

Intuition frames that help you reason about this concept fast — especially under pressure.

Model 1: The Assigned Cashier

Imagine a supermarket where each shopper's tab is stored in their assigned cashier's personal notepad. The load balancer (store manager) assigns each new shopper to a specific cashier (sticky session). If that cashier goes on break (node failure), the tab is gone — the other cashiers know nothing about it. If the store adds new cashiers, the manager doesn't reassign existing shoppers — new cashiers stand idle while the original ones are overloaded. The fix: move all tabs to a central register system (Redis) so any cashier can look up any shopper's tab. This captures loss-on-failure, uneven load, and the correct fix in one frame.

Model 2: Reserved Parking vs. Pay-and-Display

Cookie-based affinity is a reserved parking spot — a valet gives you a ticket (cookie) guaranteeing your car (session) goes to space B-7 every time. IP-hash is always assigning parking level by license plate region — works for individual cars but if a bus parks (corporate IP with 5,000 users), the entire fleet floods one level. The "correct fix" framing: stateless services are self-driving cars that park anywhere and know where they are without a reserved space. This model breaks down for session draining mechanics but is ideal for the IP-hash NAT problem.

Model 3: The Sticky-Note Anti-Pattern

Think of server-side session state as sticky notes plastered on one server's monitor. Sticky sessions are the rule "always send this user back to the server with their notes." If that server burns down, the notes are gone. If you add a new monitor (server), you can't move notes automatically. The right answer isn't better rules about who can read whose sticky notes — it's to stop using sticky notes and use a shared whiteboard (Redis/JWT) that anyone can read. This model makes the anti-pattern nature of stickiness viscerally clear.

09 ⚙️ How It Works — Mechanics

Step-by-step explanation of the internal mechanism.

Cookie-Based Affinity — Happy Path

1. Client sends first request; no affinity cookie present yet.
2. LB selects backend via normal algorithm (e.g., round robin) → assigns B2.
3. LB inserts: Set-Cookie: AWSALB=<encoded_B2_id>; Path=/; Max-Age=86400 into HTTP response.
4. Client stores cookie; sends it on every subsequent request.
5. LB reads cookie, extracts B2 ID, routes directly to B2 — bypassing normal algorithm.
6. On B2 failure: LB detects via health check, removes B2. Requests with stale cookie are rerouted to a new backend. Session state is lost unless externalized to Redis.

IP-Hash Affinity — Mechanics and Failure

LB computes hash(source_IP) % num_backends → deterministic routing, no cookie needed. On node count change (add/remove), the modulus changes and all hash mappings recompute — causing mass session disruption. Under corporate NAT: thousands of users share one public IP, all hashing to the same backend → hot node, defeats the purpose of load balancing entirely.

Session Draining During Deployment

Operator marks B2 as "draining." LB stops routing new sessions to B2 but continues honoring existing affinity cookies pointing to B2. LB monitors active connection count on B2; once zero (or drain timeout expires), B2 is removed. Sessions that didn't complete gracefully are dropped. Planned deployments work; sudden crashes do not — there's no draining on an unexpected failure.

The Hidden Single Point of Failure

With stickiness, each user's session availability equals the availability of their pinned node (e.g., 99.9% for one server). Without stickiness, each request can hit any of N nodes — effective session availability is approximately 1 − (0.001)^N ≈ 100%. Stickiness silently degrades user-level reliability even when the overall cluster is highly available.

10 🏭 Real-World System Examples

Where does this appear in production systems you know?
SystemHow This Concept AppliesNotes
AWS ALB Duration-based sticky sessions via AWSALB cookie; configurable TTL (1 sec – 7 days). Application-based stickiness uses a custom cookie from the target. AWS docs recommend session stores for stateless design instead; stickiness is explicitly called a scaling limitation.
HAProxy Implements stickiness via stick-table (cookie-based) and balance source (IP-hash). Widely used in on-prem and self-hosted deployments. NAT collapse applies to balance source; stick on cookie is more robust for public internet traffic.
NGINX ip_hash directive pins clients to upstreams. Cookie-based sticky requires NGINX Plus or Lua module in open-source. Open-source NGINX's ip_hash is the most common entry-point into the NAT collapse gotcha.
Legacy Java EE / PHP Tomcat JSESSIONID, PHP PHPSESSID stored in server memory — the canonical motivating use case for stickiness. Modern containerized versions externalize sessions to Redis; stickiness during migration is the acceptable stopgap.
WebSocket connections WebSocket upgrades require all frames of a session to traverse the same TCP connection to the same backend. LBs must maintain affinity for connection lifetime. This is a legitimate, unavoidable use case — the connection itself is stateful at the transport layer. Not an anti-pattern.

11 ⚖️ Trade-offs

Every design decision has a cost. What are you giving up?
✅ Benefit❌ Cost / Limitation
Zero application code change — works with existing stateful apps New backend nodes don't absorb existing sticky traffic — scaling benefit is limited to new sessions only
Simple to configure at the LB level (one flag or directive) Node failure drops all sessions pinned to that node — no redundancy at the session level
No external state store required — avoids Redis operational overhead Load distribution skews over time as active users cluster on specific nodes
Works for WebSocket transport-layer affinity (protocol requirement) Increases deployment complexity: session draining required for rolling deploys; no instant blue/green switch
Buys migration time from stateful to stateless architecture IP-hash variant collapses under NAT — silently creates a hot node with no warning

12 🎯 Interview Application

How do you use this concept in a design interview? What triggers it?

When an interviewer asks / says:

  • "How would you handle user sessions across multiple servers?"
  • "What happens if a backend node goes down mid-session?"
  • "Why can't you just add more servers to handle the load?" (in a context implying server-side state)
  • "The app currently uses in-memory session storage — how do you scale it?"

What you say / do:

Proactively flag sticky sessions as a smell: "If this service stores session state server-side, we'd need sticky sessions to function correctly — but that limits our scaling. I'd prefer to externalize session state to Redis so any node can serve any request, which gives us true horizontal scalability and fault tolerance." Frame it as a design decision you're consciously making, not a given — show you recognize the trade-off and know the superior alternative.

The trade-off statement (memorize this):

"If we choose sticky sessions, we preserve compatibility with server-side session state without refactoring, but we pay in reduced horizontal scalability and no session-level redundancy — a node failure drops all active sessions on that node. For a new system, the right call is to externalize state to Redis and design stateless services; for a legacy migration, sticky sessions can be an acceptable interim while we refactor."

13 ⚠️ Common Misconceptions & Gotchas

What do candidates get wrong? What nuance is the interviewer probing for?
❌
Misconception: Sticky sessions give you session redundancy — if one server fails, the other servers pick up the sessions.
✅
Reality: Stickiness is about routing, not replication. Sessions pinned to a failed node are simply lost. The other nodes know nothing about those sessions. True redundancy requires externalizing state.
❌
Misconception: IP-hash affinity is more reliable than cookie-based because it doesn't depend on client cookies.
✅
Reality: IP-hash is fragile under NAT/CGNAT — thousands of users behind one public IP all hash to one server, creating a hot node. Cookie-based affinity is generally more robust and widely preferred for HTTP workloads.
❌
Misconception: Adding more backend nodes while using sticky sessions scales performance proportionally for existing users.
✅
Reality: New nodes only absorb new sessions. Existing sessions remain pinned to their original nodes until they expire or those nodes restart. The scaling benefit is limited to net-new traffic only.
❌
Misconception: WebSocket stickiness is the same anti-pattern as HTTP session stickiness and should be eliminated the same way.
✅
Reality: WebSocket stickiness is a protocol requirement — a WebSocket upgrade and all subsequent frames must use the same TCP connection to the same backend. This is unavoidable. Redis Pub/Sub handles message fan-out, but the persistent TCP connection still needs affinity.
