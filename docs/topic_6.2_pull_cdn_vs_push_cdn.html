6.2 Pull CDN vs. Push CDN
Content Delivery Models — Demand-Driven vs. Pre-Loaded

§0 🗺️ Topic Overview

Pull CDN and Push CDN are the two fundamental models for getting content onto edge servers. The central tension is control vs. automation: Pull CDN is origin-driven on demand (the edge fetches on first miss), while Push CDN is operator-driven upfront (you explicitly upload content to all PoPs before any request arrives). Mastering this topic means understanding which model fits your content's update frequency, predictability, and volume — and being able to justify the choice without hand-waving.

  • 1. The Pull cache fill flow — On a cache miss, the edge fetches from origin and caches the response. Understand the cold-start implication: the first user per PoP always gets origin latency. This is the core UX risk of Pull CDN.
  • 2. The Push pre-load flow — You control what lives at the edge. Files are uploaded proactively, so first request = cache hit. But you own the entire lifecycle: upload, invalidate, and cleanup.
  • 3. TTL and invalidation in Pull CDN — Short TTL = fresh, high origin load; long TTL = stale, low origin load. Know how to break this tension with versioned URLs or purge APIs.
  • 4. Storage and cost tradeoffs — Push CDN pre-populates every PoP regardless of demand; Pull CDN stores only what is requested at each location. At scale, Push storage cost = files × PoP count.
  • 5. When each model fits — Pull = dynamic/unpredictable demand; Push = large, static, predictably global. Real production systems typically use a hybrid.

§1 🎯 Goal of This Subtopic

§5 🧠 Core Definition

Pull CDN is a demand-driven model where edge servers cache content on the first cache miss by fetching from origin; Push CDN is an operator-driven model where content is proactively uploaded to edge servers before any user request, guaranteeing zero cold-start latency at the cost of managing the full content lifecycle manually.

🔵 Pull CDN Demand-Driven

  • Edge fetches from origin on cache miss
  • Self-populating — no operator action needed
  • Cold-start penalty on first request per PoP
  • TTL controls freshness window
  • Storage efficient — only requested content stored

🟠 Push CDN Pre-Loaded

  • Operator uploads content before requests arrive
  • Zero cold-start — always a cache HIT
  • Operator manages upload, update, delete
  • Storage cost = files × PoP count
  • Best for large, stable, globally demanded files

§6 📦 Core Concepts

Pull CDN — Demand-Driven Caching

An edge server that receives a request for uncached content fetches it from origin, caches it locally, and returns it to the user. The duration of caching is controlled by Cache-Control: max-age (or s-maxage for CDN-only TTL). The first request per PoP per key always goes to origin — this is the cold-start penalty. Pull CDN is self-maintaining: as TTLs expire, the edge automatically refreshes on the next miss. This makes Pull ideal for content that changes frequently or has unpredictable geographic demand.

Push CDN — Proactive Pre-Loading

Operators explicitly upload content to CDN edge nodes before any user requests it. The CDN stores the files and serves them directly — there is no origin fetch on demand, so the first request is always a cache HIT. However, the operator must manage the full content lifecycle: pushing updates, issuing deletes when content is retired, and monitoring storage usage. Push is appropriate for large, rarely-changed assets where you can predict global demand (e.g., game clients, software binaries, marketing campaign videos).

TTL and Cache Invalidation (Pull-specific)

TTL (Time-To-Live) controls how long Pull CDN edges serve a cached copy before considering it stale. The three invalidation strategies have different trade-offs:

  • TTL expiry — simplest, but content may be stale for the full TTL window after an update
  • Versioned URLs — app.abc123.js bypasses cached old versions entirely; no purge needed; safest approach
  • CDN Purge API — forcibly evicts a key from all PoP caches; propagates in 1–10s globally; charged at scale (CloudFront: $0.005/path after first 1000/month)

Push CDN Storage Economics

Push CDN requires reserving edge storage at every targeted PoP. With 200–500 global PoPs, storage costs multiply fast. Pushing a 1 GB binary to 300 PoPs consumes 300 GB of edge storage immediately — regardless of whether users in those PoPs actually request it. Pull CDN avoids this by only storing what is requested. This is the primary scaling constraint of Push CDN: it works well for a small library of large files, but is prohibitively expensive for thousands of small, frequently updated files.

Hybrid CDN

Most production systems use both models simultaneously. Static, large, rarely-changed assets (JS/CSS bundles, images, video assets) are served via Pull CDN with long TTLs. Large binary distributions (software installers, game updates) may use Push CDN. Dynamic content (API responses, personalized pages) typically bypasses CDN entirely or uses very short TTLs.

§7 🔍 First Principles — Why Does This Exist?

Before CDNs, every user request traversed to origin regardless of geographic distance. A user in Tokyo requesting a file from a US-hosted server would see 150+ ms of round-trip latency — and origin had to serve every user globally, creating both a latency problem and a capacity problem.

CDNs solved this by distributing content to edge nodes closer to users. But this immediately raised the question: when should content move to the edge? Two philosophies emerged:

Pull CDN answers: "move it when someone asks" — demand drives population organically. This works perfectly for unpredictable content — you can't know which PoPs will see demand for a given user avatar or article. Fetching on demand avoids pre-populating 300 PoPs with content most of them will never serve.

Push CDN answers: "move it before anyone asks" — trade edge storage cost for guaranteed availability. This works for predictable, large content — if you're shipping a game installer, you know millions of users worldwide will request it immediately at launch. Waiting for the first miss per PoP when 1 million users hit simultaneously would be catastrophic. Pre-loading is the only viable strategy.

Both models co-exist because different content types have fundamentally different demand properties, and no single model is globally optimal.

§8 🗺️ Mental Models

Model 1: The Public Library vs. the Bookstore

Pull CDN is like a public library — books are brought to the local branch only when patrons request them. The first patron who requests a rare book waits for it to arrive (cold start); subsequent patrons find it already on the shelf. Push CDN is like a bookstore that pre-stocks shelves before opening — every title is on the shelf the moment doors open, but someone made buying decisions upfront and unsold stock sits on shelves (wasted storage). The model breaks down in that CDNs have near-instant distribution globally; use it to understand the demand-driven vs. supply-driven mental frame.

Model 2: Cache-Aside vs. Cache Pre-Warming

Pull CDN is architecturally identical to the cache-aside pattern — the edge is the cache, origin is the database, TTL is the expiry. This is the same pattern as application-layer caching, just pushed to the network edge. Push CDN maps to cache pre-warming — explicitly seeding the cache before traffic arrives. If you understand cache-aside (Topic 5.1) and pre-warming, you already understand the CDN analogs. The key difference: cache-aside usually has one shared cache; Pull CDN has 200–500 PoP caches globally, each with independent cold-start.

Model 3: The Freshness Dial

Think of Pull CDN TTL as a dial between two extremes: TTL=0 means "always hit origin" (CDN provides connection offload only, no caching benefit); TTL=∞ means "never update at edge" (maximum performance, broken for changing content). Every Pull CDN configuration is finding the right position on that dial for a given content type. Where the model breaks down: TTL is set per Cache-Control header on the origin response, not a global CDN setting — misconfigured origin responses result in unintended positions on the dial without any visible CDN-side error.

§9 ⚙️ How It Works — Mechanics

📡 Pull CDN — Cache Miss & Cache Hit Flows

Pull CDN: cache miss triggers origin fetch (cold start); cache hit serves from edge with no origin contact

📤 Push CDN — Upload Lifecycle

Push CDN: operator uploads once; CDN distributes to all PoPs; every user request is a guaranteed cache HIT

⏱️ Pull CDN Invalidation Strategy Decision

Invalidation strategy decision tree for Pull CDN — TTL expiry → versioned URLs → Purge API

Pull CDN — Key HTTP Headers

Cache-Control: max-age=86400 — cache for 24h at both CDN and browser. Cache-Control: s-maxage=86400 — cache at CDN for 24h but browser respects its own cache independently. Cache-Control: no-store — never cache, defeats CDN entirely. Vary: Accept-Encoding — creates separate cache entries per encoding (gzip vs. br); can fragment cache and reduce hit rates if overused. Surrogate-Control: max-age=N — Fastly/Varnish CDN-only TTL not forwarded to browser.

Push CDN — Lifecycle Details

Upload via CDN API (REST or rsync). CDN distributes to all target PoPs immediately. Serve: any request = guaranteed HIT. Update: re-upload new version to same path or new path. Delete: explicit delete API call; storage freed after global confirmation. Storage is charged for all uploaded content across all PoPs until deletion, regardless of request volume.

§12 🎯 Interview Application

When an interviewer asks / says:

  • "How would you serve static assets globally with low latency?"
  • "You're launching a new game — how do you ensure players download the client fast worldwide?"
  • "Our homepage CSS/JS takes 2s to load for users in Asia. What do you do?"
  • "How would you handle a massive simultaneous traffic spike on a new product launch?"

What you say / do:

During high-level design, when proposing a CDN layer, immediately distinguish which model applies to the specific content type. For static web assets: "We'd use Pull CDN with long TTLs and versioned filenames — deploy updates with a new hash, no manual purge needed." For large binary distributions: "We'd use Push CDN to pre-load content at all major PoPs before the launch — we can't afford first-request origin latency when millions of users hit simultaneously."

"If we choose Pull CDN, we get automatic edge population and storage efficiency, but we pay with cold-start latency for the first request per PoP and a dependency on correct Cache-Control headers. For this system — which has unpredictable demand geography and content that changes frequently — Pull is the right call because the cold-start penalty is rare and manageable."

§13 ⚠️ Common Misconceptions & Gotchas

❌
Misconception: CDN always caches content — just put assets behind a CDN and they'll be served from the edge.
✅
Reality: Pull CDN only caches if the origin returns a cacheable Cache-Control header. An origin responding with Cache-Control: no-store or no cache header is not cached at all — every request hits origin regardless of CDN presence. Always verify origin response headers.
❌
Misconception: Push CDN is always better because there's no cold start.
✅
Reality: Push CDN multiplies storage cost by the number of PoPs. For thousands of small, frequently updated files (e.g., user avatars, article thumbnails), Push is operationally infeasible and expensive. Pull handles large-scale dynamic content libraries far more efficiently.
❌
Misconception: Cache invalidation in Pull CDN is instant — issue a purge and all users get fresh content immediately.
✅
Reality: CDN purge APIs propagate globally in 1–10 seconds (not instant), and at scale purge APIs are rate-limited and charged per path. Versioned URLs are more reliable for guaranteed freshness because they don't rely on purge propagation timing.
❌
Misconception: Pull and Push CDN are mutually exclusive — you pick one per CDN account.
✅
Reality: Most production systems use both simultaneously within the same CDN account — Pull for regularly updated web assets, Push for large static binaries. Most CDN providers support hybrid configurations on the same distribution.
