5.7 Hot Key Problem & Mitigation

00 🗺️ Topic Overview

The hot key problem occurs when a small number of cache or data store keys attract a disproportionately large share of all traffic — saturating the single node responsible for those keys while the rest of the cluster sits idle. This is a concentration problem, not a capacity problem: adding more nodes to the cluster doesn't help because the hot key always hashes to the same shard.

The critical skill is distinguishing hot reads from hot writes and applying the right mitigation at the right layer — from local in-process caches to sharded counters and request coalescing.

📊 Diagram 1 — Hot Key: Concentration vs. Uniform Load

Consistent hashing distributes key space evenly — but if one key gets 90% of requests, its node gets 90% of traffic. More nodes don't help.

  • 1. Why consistent hashing alone doesn't solve itConsistent hashing distributes keys by name hash, not access frequency. A viral post always lands on one shard. Adding more shards doesn't change which shard product:42 lands on.
  • 2. The three mitigation familiesLocal L1 caching (eliminates network round-trips entirely), key replication (spreads reads across N copies), request coalescing (serializes concurrent fetches within a process). Each solves a different failure mode.
  • 3. Read vs. write asymmetryHot reads are easy — replicate the value. Hot writes are hard — you cannot simply replicate a write target. Sharded counters and CRDT-based designs are required.
  • 4. Detection before mitigationRedis OBJECT FREQ + LFU eviction, redis-cli --hotkeys, DynamoDB Contributor Insights, client-side key-rate histograms.
  • 5. Freshness vs. load trade-offEvery mitigation that adds a caching layer introduces a staleness window. The harder you push on load reduction, the more staleness you accept. Be explicit about this in every answer.

05 🧠 Core Definition

A hot key is a single key in a cache or distributed data store that receives a disproportionately large share of traffic, saturating the node or shard responsible for it while leaving the rest of the cluster underutilized — a concentration problem that no amount of horizontal scaling can solve without targeted mitigation.

06 📦 Core Concepts

Hot Key Formation

A hot key forms when access patterns are skewed — a celebrity's profile, a trending news article, a flash-sale product, a real-time leaderboard entry. The key's name consistently hashes to the same node, so all requests funnel to one machine. This is distinct from cache stampede (5.6): the key is present and warm, but the node is overwhelmed by sheer request volume.

Local (L1) In-Process Caching

Each application instance maintains a small in-memory cache (Caffeine, Guava, or a simple HashMap with expiry) holding hot keys for 1–5 seconds. Requests are served from process memory, completely bypassing the shared cache tier. With 100 app servers, the shared cache receives at most ~1% of the traffic per server. Cost: per-instance memory footprint, and up to TTL seconds of staleness with no cross-instance invalidation path.

Key Replication (Shard Cloning)

Store the hot key under N differently-named keys — e.g., product:42:r0 … product:42:r9 — distributed across N shards. Readers pick a replica at random. Writers must invalidate or update all N copies. Spreads read load across shards. Only appropriate for read-dominated hot keys; for write-heavy keys, this amplifies the write cost N×.

Request Coalescing (Singleflight)

When multiple goroutines/threads request the same key simultaneously, only the first fetches from origin; the rest block and receive the same result. Go's singleflight and similar libraries implement this. Limits origin fan-out to one request per key per process, regardless of concurrent callers. Does not help across separate processes.

Sharded Counters

For hot write keys (likes, views, inventory), writes are distributed across N counter shards: counter:post:42:shard:{rand(0,N-1)}. Reads aggregate all N shards. The count is eventually consistent during the short aggregation window. Used by Facebook for reactions, YouTube for view counts (~200 shards per video at scale).

Detection and Monitoring

Redis: set eviction policy to allkeys-lfu, then run redis-cli --hotkeys or query OBJECT FREQ key. DynamoDB: enable Contributor Insights to surface top hot partition keys. Client-side: maintain a sliding window counter per key; alert when the top 1% of keys consume >50% of traffic.

07 🔍 First Principles — Why Does This Exist?

📊 Diagram 2 — Zipf's Law: Access Distribution of Cache Keys

Real access patterns follow a Zipf-like curve. The top 1% of keys absorbs 50%+ of traffic — the entire hot key problem flows from this shape.

Distributed caches and data stores distribute keys across nodes under one assumption: access will be roughly uniform across keys. Every sharding algorithm — consistent hashing, range partitioning, hash partitioning — is designed to distribute key space evenly, not access frequency.

In practice, access patterns follow a Zipf-like power law: a tiny fraction of keys (trending content, popular users, top products) accounts for the vast majority of traffic. When this happens, the node holding the hot key becomes a single-node bottleneck inside your distributed system. Adding more nodes doesn't help — the hot key always maps to one node.

The problem is not infrastructure scale; it is concentration. Every mitigation shares one design principle: break the concentration by introducing redirection, replication, or aggregation that spreads the load across multiple nodes — or eliminates remote calls entirely.

08 🗺️ Mental Models

Model 1: The Superstar Cashier

A supermarket has 20 checkout lanes. 18 lanes have zero customers. One lane has 500 people queued because a celebrity employee is working it. Adding more checkout lanes doesn't help the 500-person queue. You fix it by: (a) letting managers handle some transactions locally — local L1 cache; (b) cloning the celebrity's workflow across 10 lanes — key replication; (c) restructuring so no single cashier handles the singular resource — sharded counters. The failure mode is concentration, not capacity.

Where it breaks down: The analogy doesn't capture the consistency cost — in a cache, "cloning" means keeping all clones in sync, which adds write overhead the supermarket model ignores.

Model 2: Reads vs. Writes — Books vs. Bank Accounts

A hot read key is like a very popular book at a library — the fix is to buy more copies (replicate). A hot write key is like a single shared bank account receiving thousands of simultaneous deposits — buying more copies doesn't help, because every copy needs to receive every write. The fix is to split the account into N sub-accounts and sum them on read (sharded counters). If you try to "replicate" a hot write key, you just amplify your write problem N-fold.

Model 3: The Power Law is the Enemy

Hot key problems are a manifestation of Zipf's law. The cure isn't a bigger node (vertical scaling); it's flattening the distribution by spreading load. Any mitigation that reduces a single node's load by a factor of N is equivalent to adding N× the capacity at that node — but without the hardware cost. However, flattening always adds latency, complexity, or staleness. You are trading one problem (saturation) for another (consistency lag, higher write amplification, or more complex code).

09 ⚙️ How It Works — Mechanics

📊 Diagram 3 — Hot Key Mitigation Decision Flow

The first question to answer is always: read-heavy or write-heavy? The answer determines the entire mitigation path.

Happy Path (No Hot Key)

Client requests GET product:42. Consistent hash maps it to node 7 of 20. Node 7 returns the cached value in <1ms. Traffic is roughly uniform across all 20 nodes.

Hot Key Failure Path

A product goes viral. 50,000 rps all request GET product:42. All 50,000 rps are routed to node 7 (same hash every time). Node 7's CPU, network bandwidth, or connection pool is exhausted. Latency for product:42 rises; other keys on node 7 also degrade (collateral damage).

Mitigation 1: Local L1 Cache

Each app server maintains a local TTL cache (e.g., 2-second TTL). On a hit, the request never leaves the process. With 100 app servers, node 7 receives at most ~500 rps (1% of 50k), plus one miss per app server per TTL cycle.

Caffeine.newBuilder().expireAfterWrite(2, SECONDS).maximumSize(1000).build()

Mitigation 2: Key Replication

Write the value to product:42:r0 … product:42:r9 across 10 shards. Readers pick product:42:r{random(0,9)}. Each shard now receives ~5,000 rps instead of 50,000. Writers must update or invalidate all 10 replicas on change. Only works when writes are infrequent relative to reads.

Mitigation 3: Request Coalescing (Singleflight)

All concurrent requests for the same key within one process share one underlying fetch. Only the first goroutine hits the cache/origin; all others block and receive the same result. Eliminates thundering herd within a single process on cache miss. Doesn't help across separate processes.

Mitigation 4: Sharded Counters (Hot Write Keys)

Write: INCR counter:likes:post:42:shard:{rand(0,N-1)}
Read: SUM(counter:likes:post:42:shard:0 … shard:N-1)

With N=100 shards, write load is distributed across 100 keys on up to 100 different nodes. Read cost increases from O(1) to O(N), but reads are cacheable and aggregate counts typically tolerate slight staleness.

Key Thresholds

  • Local L1 TTL: 1–5 seconds typical; 10–30 seconds for pure read-only reference data
  • Shard count for counters: 50–200 shards at YouTube/Facebook scale
  • Replication fan-out: 3–10 replicas per hot read key — diminishing returns above 10

11 ⚖️ Trade-offs

✅ Benefit❌ Cost / Limitation
Local L1 cache eliminates network round-trips; reduces remote cache load by factor of #app-instances Each app instance holds stale data for up to TTL seconds; no cross-instance invalidation without a pub/sub layer; per-instance memory footprint
Key replication spreads read load across N shards linearly Write amplification is N× — every update must propagate to all N replicas; consistency window during propagation; schema complexity increases
Request coalescing prevents stampede within a single process with zero added infrastructure Only works within one process; with 200 app servers, still 200 concurrent requests reaching the shared cache (one per server); adds latency for all waiters behind the single fetch
Sharded counters distribute write load linearly across N shards Read cost is O(N) — must sum all shards; count is eventually consistent during aggregation window; significantly more complex to implement and maintain
DynamoDB Adaptive Capacity handles moderate hot partition spikes automatically without code changes Has hard limits — sustained spikes above provisioned capacity still throttle; not a substitute for proper data modeling and key design

12 🎯 Interview Application

Trigger Phrases

  • "This product could get millions of views during a flash sale."
  • "We have celebrities with 100M followers — how do we handle their profile reads?"
  • "Our cache node is saturated even though we added more cache servers."
  • "How do you handle like counts on a viral post getting 500k reactions per minute?"

What You Say / Do

First, clarify whether the hot key is read-heavy or write-heavy — this determines the entire mitigation path. For a read-heavy key (product page, celebrity profile), propose local L1 caching as the first line of defense, followed by key replication if L1 alone is insufficient. For a write-heavy key (like counter, view count), propose sharded counters and explicitly acknowledge the eventual consistency trade-off. Always name the detection mechanism — how would you know you have a hot key in production?

"If we choose local L1 caching, we get near-zero remote load for hot keys, but we pay with up to N seconds of staleness per node and the inability to instantly invalidate across instances. For this system — where product data changes infrequently and slight staleness is acceptable — L1 caching is the right call. If we needed strict read-after-write consistency, we'd need a pub/sub invalidation layer, which adds operational complexity."

13 ⚠️ Common Misconceptions & Gotchas

❌Adding more shards or nodes to the cluster eliminates the hot key problem.
✅Sharding distributes key space, not access frequency. A hot key always maps to one shard regardless of cluster size. The fix is changing how the hot key is accessed — not adding infrastructure.
❌Key replication works equally well for both hot reads and hot writes.
✅Key replication helps hot reads by spreading load across N copies. For hot writes, it amplifies the write problem — every write must go to all N replicas. Hot writes require sharded counters or write aggregation.
❌Request coalescing (singleflight) solves the hot key problem at scale.
✅Singleflight only works within a single process. With 200 app servers, 200 requests still reach the cache — one per server. It prevents stampede within one process, not across processes.
❌Sharded counters are unsuitable for any production system because the count is not accurate.
✅The consistency model is configurable. Many systems (YouTube, Facebook) deliberately accept ~1 second of count staleness because exact real-time accuracy for view/like counts is not a business requirement. For inventory or financial counts, stricter patterns (centralized lock + reservation) are used instead.

1 A single product page is receiving 200,000 rps during a flash sale. Your cache cluster has 50 nodes, all at 5% utilization except one, which is at 100%. What is happening and why doesn't adding more nodes help? Show hint
💡 Think: consistent hashing maps keys by name, not by access frequency. Adding more shards distributes more keys — but product:42 still maps to exactly one shard. The hot key is a concentration problem, not a capacity problem. → Section 7
2 A social media post goes viral. It receives 500,000 like events per minute. What storage pattern would you use for the like count, and what consistency trade-off does it involve? Show hint
💡 Write-heavy key → sharded counters. Write to counter:post:42:shard:{rand(0,N)}. Read by summing all N shards. Trade-off: the count is eventually consistent during the aggregation window (~1s). This is acceptable for a like count, not for financial inventory. → Section 6 (Sharded Counters)
3 Your team proposes replicating every hot cache key across 10 shards. What question must you ask first, and under what condition does this proposal make things worse? Show hint
💡 Ask: is this key read-heavy or write-heavy? If write-heavy (e.g., a counter updated every second), replicating across 10 shards means every write must be sent to all 10 — 10× write amplification. The proposal makes the write problem worse, not better. → Section 8 (Mental Model 2)
4 Name a production system that uses sharded counters and explain how they implement read aggregation. Show hint
💡 YouTube (view counts, ~200 shards per video), Facebook/Tao (reaction counts). Read aggregation: sum all N shard keys at read time. The result is cached to avoid repeated O(N) aggregation on every view. → Section 10
5 You add a local L1 cache with a 3-second TTL to handle a hot read key. The interviewer asks: "What happens if the underlying data changes?" What is the worst-case consistency scenario, and how could you reduce the staleness window without eliminating L1 caching? Show hint
💡 Worst case: a key is updated 1ms after it's cached locally. All app instances serve stale data for up to 3 seconds. To reduce without removing L1: (a) use pub/sub (Redis Pub/Sub, Kafka) to broadcast invalidation events — app instances evict the key on receiving the event, so staleness drops from TTL to propagation latency (~ms); (b) use a shorter TTL at the cost of higher L1 miss rate. → Section 9 (Mitigation 1) + Section 11
