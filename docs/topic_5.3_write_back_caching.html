5.3 Write-Back (Write-Behind) Caching

00 🗺️ Topic Overview

What This Topic Is About

Write-back (also called write-behind) caching is the most performance-aggressive write strategy: the client is ACKed as soon as the cache write completes, and the database write is deferred to an asynchronous flush. The central tension is write latency vs. durability — you eliminate the database from the write critical path and gain coalescing benefits, but data in the dirty buffer is permanently lost if the cache node fails before flushing. Mastering this means knowing exactly when that trade is acceptable, and how production systems mitigate the durability gap.

🎯 What to Focus On

  • 1. The deferred write contract — and its durability gapThe client gets an ACK the moment the cache write completes. The database does not see this write immediately. Any data in the dirty buffer is at risk until flushed. Know this cold — it is the defining property.
  • 2. Dirty buffer managementThe cache tracks which keys are dirty (written but not yet flushed). Know how flush is triggered (time, count, eviction), and what must happen when a dirty entry is evicted — silent data loss is a common implementation bug.
  • 3. Write coalescing — the performance multiplierMultiple writes to the same key within a flush window become a single DB write. For high-frequency writes (counters, scores, session state), this can reduce DB write load by orders of magnitude.
  • 4. When write-back is justified vs. dangerousWrite-back is correct for analytics counters, game scores, and session state. It is wrong for financial transactions, inventory, and anything where a lost flush window means real-world harm.
  • 5. Durability mitigationsWAL (Write-Ahead Log), cache replication, and periodic snapshots all reduce the durability gap without eliminating the core performance benefit. Know at least two.

05 🧠 Core Definition

Write-back caching writes data to the cache and immediately acknowledges the client, then asynchronously flushes dirty cache entries to the backing database at a later time — decoupling write latency from database I/O at the cost of a durability gap during the flush window.

06 📦 Core Concepts

Deferred Write Path

The client writes to the cache and receives an ACK without waiting for the database. The database write is queued and handled by a background flush process. This makes write latency as fast as a cache write (sub-millisecond for Redis), completely decoupling the client-facing write path from the far slower database I/O.

Dirty Buffer

A dirty entry is a cache key whose in-cache value has not yet been persisted to the database. The cache maintains a dirty set or dirty flag per key. When a dirty entry is evicted before flushing, the write is silently lost — correct implementations must flush dirty entries synchronously before evicting them.

Flush Trigger Mechanisms

Write-back caches flush dirty entries based on one or more triggers: time-based (flush every N seconds), count-based (flush when dirty count exceeds a threshold), eviction-triggered (flush before evicting a dirty entry), or application-triggered (explicit flush call). Most production implementations combine time-based and count-based for predictable behavior.

Write Coalescing

Multiple writes to the same key within a flush window become a single DB write — only the latest value is persisted. For a key written 1000 times per second with a 1-second flush interval, write-back generates exactly 1 DB write instead of 1000. This is the core performance multiplier for write-heavy workloads like counters, scores, and session state.

Durability Gap

The period between a cache write and its corresponding DB flush is the durability gap. Any data written during this window is at risk if the cache node fails. The size of the gap is set by the flush interval — shorter intervals reduce risk but partially erode the performance benefit by increasing DB write frequency.

07 🔍 First Principles — Why Does This Exist?

The root problem: databases are slow relative to caches, and forcing every write to touch the database imposes unnecessary latency on write-heavy workloads. Write-through solves the stale-read problem but creates a new one: every write waits for two I/O operations, and the database becomes the bottleneck for write throughput.

For systems that are update-heavy — social media like counters, game scores, analytics event aggregators, session state — the database simply cannot absorb the write rate at acceptable latency without write-back's coalescing buffer. Write-back was invented to break this coupling: let the fast cache absorb writes at full speed, batch them, and flush to the database at a rate the DB can handle.

The underlying insight is that for many workloads, the database does not need to see every write immediately — it just needs to see the latest state eventually. Write-back trades a durability guarantee for performance headroom, and for many high-throughput systems, that trade is not just acceptable but necessary.

08 🗺️ Mental Models

Model 1: The Outbox Buffer

Think of write-back like an email outbox. You write an email and it goes to your outbox immediately — you can move on without waiting for it to send. The email client flushes the outbox in the background. If your laptop crashes before it sends, the email is gone. The outbox is the dirty buffer; "sent" is the DB flush; the crash is the durability gap in action.

Where it breaks down: Email clients typically retry on reconnect. Write-back caches may not recover dirty data on crash without explicit WAL or replication. Don't carry the analogy too far on recovery semantics.

Model 2: CPU L1/L2 Cache

Hardware CPU caches use write-back by default. When the CPU writes to a cache line, the value is held in L1/L2 and not immediately written to main memory. The dirty cache line is flushed to RAM on eviction or coherence events. Modern CPUs accept this risk because the performance gain is enormous. This is exactly the same trade-off — write-back is what makes CPU caches viable at all.

Where it breaks down: CPU caches have hardware-level coherency protocols that don't map to distributed systems. The analogy explains the concept, not the implementation.

Model 3: The Batching Multiplier

Write-back isn't just deferred writes — it's a write multiplier in reverse. Write-through writes 1 DB record per cache write. Write-back writes 1 DB record per N cache writes (where N = writes coalesced in the flush window). For counters updated 1000x/second with a 1-second flush interval, write-back reduces DB write load by 1000x. Frame it this way in interviews when justifying write-back for high-frequency writes.

09 ⚙️ How It Works — Mechanics

Normal Write Path

1. Client sends a write to the application. 2. Application writes to cache (Redis, etc.) — cache ACKs immediately. 3. Application ACKs the client — write is "done" from the client's perspective. 4. Cache marks the entry dirty. 5. Background flush process picks up dirty entries on timer or threshold trigger. 6. Dirty entries written to DB in batch (often sorted for sequential I/O efficiency). 7. On successful DB write, dirty flag cleared.

Read Path (During Flush Window)

Reads are served from cache. If the key is dirty (not yet flushed), the cache still returns the correct latest value — cache is the source of truth. The database may be stale, but reads go to cache, not DB. This is correct behavior as long as all reads and writes route through the same cache.

Eviction of Dirty Entries

If the cache runs out of space and must evict a dirty entry, the eviction handler must flush that entry to DB before eviction. Failing to do this causes silent data loss. This is a critical implementation detail — naive LRU without dirty-check handling silently drops writes. Correct write-back implementations must check dirty status and flush synchronously on dirty eviction.

Flush Failure Handling

If the DB is unavailable during a scheduled flush, the dirty buffer continues to grow. The cache must implement: retry with exponential backoff, dirty buffer size caps (reject or block new writes if buffer is full), and alerting. If the DB is down longer than memory can absorb, the cache must degrade gracefully — typically by blocking new writes or surfacing an error.

Durability Enhancement — WAL

High-stakes write-back implementations log dirty writes to a Write-Ahead Log (WAL) on durable storage before ACKing the client. On cache restart, the WAL is replayed to reconstruct dirty state. This recovers most of the durability of write-through while preserving most of the latency benefit — Cassandra's commit log + memtable is the canonical production example of this pattern.

What you say / do:

The trade-off statement (memorize this):

"If we choose write-back, we get sub-millisecond write latency and write coalescing that dramatically reduces DB load, but we pay with a durability gap — dirty writes are lost on cache failure. For this system [e.g., like counts, view counters, leaderboard scores], that trade is acceptable because eventual consistency is sufficient and we can add a WAL if we need stronger guarantees."

13 ⚠️ Common Misconceptions & Gotchas

❌Write-back means writes are eventually consistent — DB reads will be fine after a short delay.
✅During the flush window, the DB is stale. Any read bypassing the cache (a replica, direct DB query, another service) sees old data. Write-back only works correctly when all reads route through the same cache holding dirty state.
❌Write-back is just write-through with a delay — the data is safe in the cache.
✅Cache memory is volatile. A node crash, OOM kill, or hardware failure permanently loses all dirty data that hasn't been flushed. "In the cache" is not the same as "durable."
❌You can use write-back with standard LRU eviction without any special handling.
✅LRU eviction of a dirty entry without flushing it causes silent data loss. Correct write-back implementations must check dirty status before eviction and flush synchronously if the entry is dirty.
❌"Write-back" and "write-behind" are different caching strategies.
✅They are the same strategy with different names. "Write-behind" emphasizes the buffering aspect; "write-back" emphasizes the return-from-cache acknowledgment. Use them interchangeably.
