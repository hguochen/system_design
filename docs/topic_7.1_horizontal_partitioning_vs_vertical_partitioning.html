7.1 Horizontal vs. Vertical Partitioning
Splitting rows vs. splitting columns

§0 🗺️ Topic Overview

Partitioning is how you split one logical dataset across many physical stores so no single machine has to hold all the data or serve all the traffic. There are two orthogonal axes. Horizontal partitioning (sharding) splits rows — every shard has the identical schema but a disjoint subset of rows. Vertical partitioning splits columns/tables — the same entities are divided so different attributes live in different stores. They are not competitors; mature systems apply both. Mastering this subtopic means looking at an access pattern and saying precisely which axis to cut along, why, and what it costs.

  • 1. The rows-vs-columns distinction — Horizontal = same schema, different rows. Vertical = same entities, different columns. State it crisply or everything downstream is fuzzy.
  • 2. What problem each axis solves — Horizontal relieves volume & write throughput. Vertical relieves row width & hot/cold access-pattern mismatch.
  • 3. The query-locality consequence — Horizontal is cheap only when the query carries the partition key; otherwise scatter-gather. Vertical is cheap only when a query's columns live on one partition; otherwise a cross-store join.
  • 4. When you combine them — At scale: vertically decompose into tables/services first, then horizontally shard whichever table outgrows one node — then replicate each shard.
  • 5. Interview framing — "Scale writes / too many rows" → horizontal. "60 columns but reads touch 5 / blob stored inline" → vertical. Recognize the triggers.

§1 🎯 Goal of This Subtopic

After studying this, you should be able to look at any data model and access pattern and decide whether to partition horizontally, vertically, both, or not at all — and justify it by the specific pressure being relieved (row count / write QPS vs. row width / access-pattern mismatch). You should be able to draw each scheme, trace how a read and a write route under each, and name the failure mode each introduces (cross-shard scatter-gather vs. cross-partition joins).

Partitioning is the ONLY way to scale a dataset beyond the biggest single box. §5 🧠 Core Definition

Horizontal partitioning (sharding) splits a table by rows — every partition shares the identical schema but stores a disjoint subset of rows, chosen by a partition key. Vertical partitioning splits a table by columns — the same logical entities are divided so different attributes (columns or sub-tables) live in different stores, typically separating frequently-accessed narrow columns from rarely-accessed wide or bulky ones.
✂️ Cutting the Same Table Two Ways

Same table, two cuts: horizontal tears off blocks of rows; vertical slices off groups of columns

🔵 Horizontal (Sharding) split rows

Each shard = identical schema, disjoint rows. Routed by a partition key. Grows capacity + write throughput ~linearly with shard count.

⚠️ Queries without the partition key → scatter-gather across all shards.

🟠 Vertical split columns

Same entities, different attributes per store. Hot narrow columns stay cache-friendly; cold/bulky columns move out (often to object storage).

⚠️ Reads spanning both column groups → cross-store join.

§6 📦 Core Concepts

Horizontal Partitioning (Sharding)

Splitting a dataset row-wise: each shard replicates the structure but holds only a disjoint slice of the rows. A partition key (e.g., user_id) plus a strategy (hash or range) decides which shard owns each row. This is the workhorse of scaling — it grows total capacity and write throughput roughly linearly with shard count, because each shard independently absorbs a fraction of the rows and writes. The catch is routing: a read must know the partition key to go straight to the owning shard; a query without it must fan out to every shard.

Vertical Partitioning

Splitting a dataset column-wise: the same entity's attributes are divided across stores. A classic example keeps user_id, username, status, last_login (small, hot, read on every request) in one fast table, while bio, profile_image_blob, preferences_json (large, cold) live in a separate store. Vertical partitioning shrinks the hot row so more of it fits in cache and each read moves less data. It also lets you match storage engine to data shape — relational store for structured hot fields, blob/object store for the avatar. The cost: any query needing columns from both sides now performs a join across stores.

Partition Key (Shard Key)

For horizontal partitioning, the column(s) whose value determines a row's home shard — the single most consequential design choice in sharding. It must be high-cardinality and evenly distributed so load spreads uniformly, and it should align with the dominant query pattern so most queries are single-shard. A poor key (low cardinality like country, or monotonic like a timestamp) concentrates load on one shard — a hot partition.

Disjointness (Partitioning vs. Replication)

The defining property of partitioning is that each node holds different data with no overlap — as opposed to replication, where each node holds a copy of the same data. Partitioning scales capacity; replication scales read throughput and availability. They are orthogonal and almost always combined: you shard for capacity, then replicate each shard for durability and reads.

Cross-Partition Operations

The failure mode shared by both axes. In horizontal partitioning, a query lacking the partition key (or an aggregate across all rows) becomes a scatter-gather: query every shard, merge results — slow, and as expensive as the slowest shard. In vertical partitioning, a read needing columns from two partitions becomes a cross-store join — you've reintroduced the join you were trying to avoid, now across a network. Good designs keep the common-case query within a single partition.

§7 🔍 First Principles — Why Does This Exist?

A single machine has hard ceilings: finite disk (can't store more rows than fit), finite RAM (the working set must fit or you thrash), finite CPU/IOPS (only so many writes per second before disk or lock contention saturates). For a while you push those ceilings up by buying a bigger box — vertical scaling of hardware. But that curve flattens: the biggest machine is finite and its cost grows super-linearly. Eventually a dataset or a write rate simply exceeds what any one node can do.

Partitioning is the answer: split the data across many commodity nodes so aggregate capacity and throughput scale with node count. The reason there are two axes is that a node can be overwhelmed in two independent ways. It can have too many rows or too many writes — a volume/throughput problem, solved by cutting rows across machines (horizontal). Or its rows can be too wide, with a few hot columns read constantly while large cold columns bloat every row and evict useful data from cache — a width/access-mismatch problem, solved by cutting columns across machines (vertical). Both are "divide the data so no node holds all of it," but they treat different diseases — which is why a mature system uses both.

📈 One Node's Ceilings → Two Ways to Divide

A node is overwhelmed by row count/QPS OR by row width — each pressure maps to a different cut

§8 🗺️ Mental Models

Model 1: Cutting a Spreadsheet

Picture your data as one giant spreadsheet. Horizontal partitioning cuts it with horizontal lines — you tear off blocks of rows and hand each block to a different machine; every block still has all the columns. Vertical partitioning cuts it with vertical lines — you slice off groups of columns and put each group on a different machine; every machine still has all the entities but only some of their fields. Where it breaks down: it shows the shape of the split but not routing or the join cost — the runtime consequence lives elsewhere.

Model 2: The Library Analogy

A library that outgrows one building can expand two ways. Horizontal: open a second identical building and split books by call number (A–L here, M–Z there) — same layout, different books; to find a book you must know which building (the partition key). Vertical: keep frequently-browsed items (catalog, new releases) in the fast front room and move rarely-touched archives (bound periodicals, microfilm) to a distant annex — same items conceptually, split by access frequency. A patron needing both a new release and its archived companion makes two trips (a cross-partition read). Where it breaks down: if you push the "same items" mapping too literally.

Model 3: Volume vs. Width Diagnosis

When you feel scaling pain, ask one diagnostic question: is the problem how MANY rows, or how WIDE the rows? "Many" (row count, write QPS, total bytes) → cut horizontally. "Wide" (60 columns, giant blobs inline, hot/cold mixed) → cut vertically. This turns an open-ended "how do we scale this?" into a fast, defensible decision. Where it breaks down: at large scale the honest answer is often "both" — the frame is a starting diagnosis, not the final architecture.

§9 ⚙️ How It Works — Mechanics

🧭 Horizontal Routing — Single-Shard vs. Scatter-Gather

The partition key decides everything: with it, one hop; without it, a fan-out to every shard

Horizontal — Read Path

Single-shard (happy case): the app issues a query keyed by the partition key (e.g., WHERE user_id=12345). A routing layer (client library, a proxy like Vitess/ProxySQL, or a coordinator) applies the partition function — hash or range — to compute the owning shard, sends the query to that one shard (and its read replica), and returns. Cost ≈ a single-node query. Scatter-gather (failure mode): a query lacking the partition key or aggregating across all rows can't be narrowed to one shard, so the router fans it out to all N shards in parallel, each computes a partial result, and the coordinator merges. Latency is bounded by the slowest shard and gets more expensive as you add shards — which is why the partition key must align with the dominant query pattern.

Horizontal — Write Path

Compute the shard from the partition key of the row being written, write to that shard's primary, and let the shard's own replication propagate to its replicas. Writes to different keys land on different shards, so aggregate write throughput scales with shard count — the core win.

Vertical — Read & Write Paths

A hot-path read (render a user's header) touches only the narrow hot table: small, cache-friendly rows. A read needing cold columns (open the full profile) queries the second store for bio, avatar — a second lookup or application-level join. Writes to hot fields touch only the hot store; writes to cold fields touch only the cold store. A single logical update spanning both stores becomes a multi-store write — either accept eventual consistency between them or coordinate with a transaction/saga, the price of the split.

⚠️ Partition key sizing & skew: Choose a high-cardinality, evenly-distributed key that matches your queries. Size shard count as shards ≈ max( ceil(data / per_shard_capacity), ceil(peak_write_qps / per_shard_write_capacity) ) with growth headroom. A low-cardinality or monotonic key (country, boolean, timestamp) concentrates load on one shard — a hot partition (see §7.6).

§11 ⚖️ Trade-offs

⚖️ Horizontal vs. Vertical — Key Dimensions

They relieve different pressures — horizontal multiplies throughput; vertical isolates access patterns

✅ Benefit❌ Cost / Limitation
Horizontal: scales total capacity and write throughput ~linearly with shard count Cross-shard queries (no partition key, or aggregates) become scatter-gather, bounded by the slowest shard
Horizontal: each shard is smaller, so per-node indexes, backups, and working sets stay manageable Requires a routing layer and a well-chosen partition key; a bad key creates hot shards and skew
Vertical: shrinks the hot row so more fits in cache and each read moves less data Reads spanning both partitions become cross-store joins, adding latency and code complexity
Vertical: lets you match storage engine to data shape (relational hot fields, blob store for media) A single logical update spanning partitions loses single-store transactionality (eventual consistency or sagas)
Both: partitioning is the only way to hold a dataset larger than the biggest single node Partitioning alone gives no availability — each partition is a SPOF until separately replicated; op complexity rises

§12 🎯 Interview Application

Trigger Phrases

  • "This table won't fit on one machine / has billions of rows — how do you scale it?"
  • "Write throughput is saturating the primary database."
  • "The row is very wide but the hot path only reads a few columns."
  • "You're storing large documents/images inline with the record — is that a problem?"
  • "What's your partition key, and what happens to queries that don't use it?"

What You Say / Do

Diagnose the axis first, out loud: "Is this a volume problem — too many rows or too much write QPS — or a width problem — rows too wide with hot and cold columns mixed?" Then commit: for volume, "I'd horizontally shard by <key> so capacity and writes scale with shard count, and I'd pick the key to match the dominant query so most reads are single-shard." For width, "I'd vertically split the hot narrow columns from the cold/bulky ones — moving blobs to object storage — so the hot table stays cache-friendly." Then pre-empt the cost: name scatter-gather (horizontal) or cross-store joins (vertical) before the interviewer does, and note that each shard still needs replication for availability.

💬 Trade-off statement (memorize this):

"Horizontal sharding buys me linear capacity and write scaling, but only if queries carry the partition key — otherwise I pay scatter-gather across all shards. Vertical partitioning buys me a lean, cache-friendly hot path, but any read spanning both column groups becomes a cross-store join. For this system, since reads are almost always keyed by user_id and the table is uniformly accessed, this is a volume problem — I'd shard horizontally by user_id and push cross-user analytics to a separate OLAP store rather than scatter-gathering."

§13 ⚠️ Common Misconceptions & Gotchas

❌
Misconception: Partitioning and replication are the same thing, or partitioning gives you high availability.
✅
Reality: They are orthogonal. Partitioning splits different data across nodes to scale capacity; replication copies the same data to scale reads and survive failures. A partition with no replica is a single point of failure for its slice of data. Real designs shard and replicate.
❌
Misconception: "Vertical partitioning" means the same as "vertical scaling."
✅
Reality: Vertical scaling means making one machine bigger (more CPU/RAM). Vertical partitioning means splitting a table's columns across stores. Same word, unrelated concepts — mixing them up is a classic interview stumble.
❌
Misconception: Once you shard horizontally, all queries get faster.
✅
Reality: Only queries carrying the partition key get faster (single-shard). Queries without it — aggregates, secondary-attribute lookups — get slower, because they fan out to every shard and wait on the slowest one. The partition key must match your dominant access pattern or you've hurt the common case.
❌
Misconception: The sharding mechanism (hash vs. range) is the main decision.
✅
Reality: The partition key choice matters more. A low-cardinality or skewed key (country, boolean, monotonic timestamp) creates hot shards regardless of hash or range. Pick a high-cardinality, evenly-distributed key that aligns with queries first; the mechanism is secondary.

1
Define horizontal and vertical partitioning in one sentence each, and state the single distinguishing property between them.
▼ Hint
💡 Horizontal = split by ROW (identical schema, disjoint rows via a partition key). Vertical = split by COLUMN/TABLE (same entities, different attributes per store). Distinguishing property: horizontal keeps all columns and splits rows; vertical keeps all entities and splits columns. Rows vs. columns. Revisit §5.
2
A users table is 60 columns wide; the hot path reads only 4 of them, but each row also stores a 2MB inline avatar blob that's rarely read. Which partitioning axis applies, and what specifically do you do?
▼ Hint
💡 A WIDTH problem → VERTICAL. The 2MB blob bloats every row and evicts useful data from cache. Move the avatar to an object store (S3/GCS) keyed by user_id and split the hot 4 columns into a narrow users_core table. Hot path now reads small rows; the rare full-profile read pays a second lookup. Revisit §6 and §8 (Model 3).
3
You horizontally shard users by user_id. A new feature needs SELECT COUNT(*) FROM users WHERE country='JP'. What happens, and why is it slow?
▼ Hint
💡 The query has no partition key (filters by country, not user_id), so the router can't target one shard — it scatter-gathers to ALL shards, each computes a partial count, and the coordinator sums them. Slow because latency is bounded by the SLOWEST shard and cost grows with shard count. Fix: serve such analytics from a separate OLAP store / secondary index by country. Revisit §9.
4
Name a real system and describe how it applies one of these axes, including the routing or join consequence.
▼ Hint
💡 Vitess over MySQL (YouTube): horizontally partitions tables across many MySQL instances behind one logical DB; its router resolves single-shard (has the key → one instance) vs. scatter-gather (fanned out and merged). Alternatives: Instagram shards Postgres by user ID; DynamoDB/Cassandra partition by key hash. Revisit §10.
5
A candidate says "we'll shard the table, which also makes us fault-tolerant since data is spread across nodes." What's wrong, and what's missing?
▼ Hint
💡 Wrong: partitioning is NOT replication and provides no availability. Spreading DIFFERENT data across nodes means each shard is a single point of failure for its slice — lose a shard, lose that slice. Missing: REPLICATION under each shard. Correct design shards for capacity AND replicates each shard for durability/reads. Orthogonal and combined. Revisit §6 and §13.
