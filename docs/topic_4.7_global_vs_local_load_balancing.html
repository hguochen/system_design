4.7 Global vs. Local Load Balancing

00 🗺️ Topic Overview

What This Topic Is About

Global vs. local load balancing is the architectural distinction between routing traffic within a single data center (local) and routing traffic across multiple data centers or cloud regions worldwide (global). Local LB — covered in 4.1–4.6 — distributes requests across a server fleet within one region. Global LB sits one tier above: it decides which region a user's request should reach before local LB takes over. The central tension is between latency reduction and fault isolation on one side, and the operational complexity of cross-region health steering and data locality on the other.

🎯 What to Focus On

  • 1. The mandatory two-tier hierarchyGlobal LB selects the region; local LB selects the server within that region. Every globally distributed system has both. Be able to draw and explain this stack end-to-end in an interview.
  • 2. DNS-based geo-routing vs. anycastDNS routing returns different IPs based on the client's geolocation or measured RTT. Anycast assigns one IP to many PoPs and uses BGP routing to reach the nearest one. Know how each works, what layer it operates at, and its failure modes.
  • 3. DNS TTL failover lag — the #1 gotchaWhen a region fails, DNS-based routing can't redirect clients until their cached TTL expires. Real-world lag is often 2–5× the configured TTL due to resolver non-compliance. This is the most probed follow-up on global LB.
  • 4. Session and data locality after global failoverIf a user's session data lives in EU-West and global LB redirects them to US-East, their session is gone. Stateless design (JWTs) or global session replication is required to survive regional redirects cleanly.
  • 5. Health checking at region granularityGlobal LBs probe regional endpoints, not individual servers. Threshold-based failover (e.g., 3 of 3 checkers agree) prevents flapping. Understand the detection window: check interval × failure threshold.

05 🧠 Core Definition

What is it, in one sentence?

Local load balancing distributes traffic across servers within a single data center using L4/L7 mechanisms; global load balancing sits above this layer and steers traffic across multiple geographically distributed data centers or cloud regions — typically via DNS-based geo-routing or anycast — to minimize latency, absorb regional failures, and distribute capacity across the globe.

06 📦 Core Concepts

The essential building blocks — terms and ideas you must have solid before going deeper.

Local Load Balancing

Local LB operates within a single data center or availability zone, distributing requests across a fleet of identical servers using algorithms like round robin, least connections, or IP hash. It operates at L4 (TCP/UDP) or L7 (HTTP) and makes decisions in microseconds based on per-server health and load. Local LB is unaware of geography — it assumes all backends are reachable and roughly equivalent in cost. Everything in subtopics 4.1–4.6 describes local LB.

Global Load Balancing (GSLB)

GSLB routes incoming traffic to one of several regional clusters worldwide before local LB takes over. Its decisions are coarser — select the US-East cluster, not server-7 within US-East. GSLB health checks operate at the region level: it probes a regional virtual IP or health endpoint and marks the entire region healthy or unhealthy. The two primary mechanisms are DNS-based routing and anycast.

DNS-Based Geo-Routing

The DNS resolver returns different A/AAAA records based on the requesting client's location or measured RTT to each region. AWS Route 53 supports latency-based routing (return the IP of the region with lowest measured RTT to the client), geolocation routing (return the IP for the geographically closest region), and failover routing (primary/secondary with health check integration). Failover is bounded by DNS TTL — clients cache the DNS response and continue routing to the old IP until the TTL expires, even if the health check has already failed the region.

Anycast

Anycast assigns the same IP address to servers in multiple locations. BGP routing naturally delivers packets to the topologically nearest instance of that IP. When a PoP goes down, BGP withdraws the route and packets are automatically redirected to the next-nearest PoP — typically within milliseconds. Cloudflare, Google's DNS (8.8.8.8), and large CDNs use anycast extensively. The trade-off: anycast requires BGP control (your own AS and IP block), making it complex to operate and inaccessible to most application teams directly.

Two-Tier Traffic Architecture

Every globally distributed system uses a two-tier model: (1) Global LB steers the client to the correct region, and (2) Local LB within that region distributes the request across servers. The global tier is coarse, health-check-driven, and geography-aware. The local tier is fine-grained, server-aware, and latency-optimized. Confusing the two layers in an interview is a common mistake — be explicit about which tier you're describing at each step.

Region-Level Health Checks

Global LBs do not health-check individual servers — that's the local LB's job. GSLB health checks probe a regional endpoint (e.g., /healthz on a regional load balancer VIP) and mark the entire region in or out. Threshold-based failover (e.g., fail after 3 consecutive check failures) prevents flapping during transient blips. Check interval × failure threshold = maximum detection window before failover begins: e.g., 30s × 3 = 90s.

07 🔍 First Principles — Why Does This Exist?

What fundamental problem does this concept solve? Why was it invented?

A single local load balancer — even a highly available, redundant one — is physically located in one data center. Two fundamental problems forced global load balancing into existence:

Latency is physics. Light travels ~200km/ms in fiber. A user in Tokyo making a request to a US-East data center experiences ~150ms of irreducible round-trip latency just from the speed of light, before the application processes anything. For latency-sensitive applications, this is unacceptable. You must place compute near users — and then you need a mechanism to route users to the nearest compute.

Single-region failure is catastrophic. A natural disaster, power grid failure, cloud provider outage, or network partition can take an entire data center offline. With a single region, this means total application unavailability. With multiple regions, you need a mechanism that detects the regional failure and redirects traffic to surviving regions — automatically and quickly.

Local LB only solves within-region distribution. It cannot reason about geography or cross-region failover. GSLB was invented specifically to add the geographic routing layer that local LB was never designed to handle.

08 🗺️ Mental Models

Intuition frames that help you reason about this concept fast — especially under pressure.

Model 1: Airport Hub-and-Spoke

Think of global LB as the international flight booking system and local LB as the ground transport network within each city. When you book a flight from Singapore, the booking system (GSLB) decides you'll fly into New York JFK — not London or Tokyo. Once you land at JFK, the ground transport (local LB) gets you to your specific hotel. The booking system has no idea which hotel you're going to — that's not its job. It just picks the right city. If JFK closes due to a storm (region failure), the booking system reroutes your flight to Chicago. This captures the two-tier hierarchy and regional failover cleanly. It breaks down for anycast — there's no geographic pre-assignment; BGP just routes you to whichever "airport" is nearest at the routing level.

Model 2: DNS TTL as a Stale Road Map

Imagine your GPS downloaded a road closure map at midnight (DNS resolution). You're navigating at 6am using that stale map. If a highway closed at 3am (region went down), your GPS still routes you that way until you re-download the map (DNS TTL expires and you re-resolve). Short TTLs = more frequent map downloads = fresher data, but more DNS query load. Many GPS systems (resolvers) also ignore the "expiry time" on the map and cache even longer than instructed — which is why real failover lag is 2–5× the configured TTL. This makes the TTL problem viscerally concrete.

Model 3: Anycast as Campus Wi-Fi Roaming

Anycast is like enterprise Wi-Fi roaming — you have one network SSID (one IP address), and whichever access point is nearest connects your device automatically as you walk around campus. If one access point fails, your device seamlessly connects to the next nearest one without any action on your part. There's no "check which access point is closest" step — the network handles it at the routing layer. This captures the BGP-level transparency and sub-second failover of anycast. It breaks down when explaining why not every system uses anycast — you need BGP control and your own IP block, which requires significant network infrastructure that most application teams don't own.

09 ⚙️ How It Works — Mechanics

Step-by-step explanation of the internal mechanism.

DNS-Based Geo-Routing — Happy Path

1. User in Tokyo types api.example.com. Their device queries their local DNS resolver.
2. The resolver queries the authoritative DNS (e.g., Route 53). Route 53 checks the client resolver's IP geolocation or measured RTT to each regional endpoint.
3. Route 53 returns the IP of the Tokyo regional cluster (or nearest available healthy region). TTL is set to 60 seconds.
4. Client connects to the Tokyo regional IP → hits the regional load balancer → local LB distributes to a healthy server in the Tokyo cluster.
5. Client caches the Tokyo IP for 60 seconds; all requests within that window bypass DNS entirely.

DNS-Based Geo-Routing — Regional Failure

1. Tokyo cluster fails health checks (e.g., 3 consecutive failures over 30s = 90s detection lag).
2. Route 53 marks Tokyo as unhealthy and begins returning the Singapore IP for new DNS queries.
3. Clients who already resolved Tokyo continue hitting it for up to their remaining TTL. With TTL=60s, worst case is 60s of failed requests being sent to the dead region before re-resolution kicks in — and resolvers may cache longer.
4. Client re-resolves → gets Singapore IP → traffic redirects. Total failover lag: health check detection time + TTL expiry.

Anycast — Routing and Failover

1. Cloudflare advertises 104.16.x.x/20 from 300+ PoPs globally via BGP. Every PoP announces the same prefix.
2. The internet's BGP routers naturally route your packets to the topologically nearest PoP — not because Cloudflare directs you, but because BGP shortest-path routing does it automatically.
3. If the Tokyo PoP goes down, it withdraws its BGP announcement. Within seconds, BGP convergence redirects packets to the next-nearest PoP (e.g., Singapore).
4. No DNS change. No TTL wait. The IP address never changes — only which physical server answers it.

Health Check Design at Global Scale

Global LB health checkers probe regional endpoints from multiple vantage points simultaneously to avoid false positives from a single checker's connectivity issue. Threshold: typically 2-of-3 checkers must agree the region is unhealthy before failover triggers. This prevents a single packet loss event from taking a region out. The check interval (10–30s) × failure threshold (3) gives the maximum detection window: e.g., 30s × 3 = 90s detection lag before failover begins.

10 🏭 Real-World System Examples

Where does this appear in production systems you know?
SystemHow This Concept AppliesNotes
AWS Route 53 Latency-based and geolocation routing policies with health-check integration for automatic regional failover. Routes clients to the nearest healthy AWS region. Most common DNS-based GSLB for AWS-hosted services; supports A/B failover between primary and secondary regions with health-check-gated switching.
Cloudflare Anycast from 300+ PoPs globally — all traffic hits the nearest PoP; local LB within each PoP forwards to origin. DDoS attacks are absorbed across all PoPs simultaneously. Anycast is why Cloudflare can absorb massive DDoS — attack traffic is distributed across all PoPs rather than concentrated on one target.
Google GCLB (Maglev) Single anycast IP fronts Google's worldwide infrastructure. Maglev software LB handles local distribution within each region. Andromeda handles the virtual networking layer. Google's Maglev paper (NSDI 2016) describes this two-tier architecture explicitly — one of the best public references on production-scale global LB design.
Netflix Open Connect DNS-based geo-routing steers clients to the nearest Open Connect Appliance (OCA) cluster. ISPs host OCAs directly in their networks. GSLB routes clients to the nearest healthy OCA cluster. Netflix supplements DNS with client-side adaptive streaming — the client SDK measures and selects the optimal server, adding a second selection layer beyond DNS.
Fastly / Akamai CDN Anycast + DNS-based hybrid — edge PoPs use anycast for initial routing, with DNS-based overrides for specific client populations or compliance regions. CDNs were the earliest adopters of GSLB; the CDN use case drove much of the operational tooling now used for general-purpose global LB.

11 ⚖️ Trade-offs

Every design decision has a cost. What are you giving up?
✅ Benefit❌ Cost / Limitation
Reduced latency — users reach the nearest region, shaving 50–200ms of RTT DNS TTL creates a failover lag window — clients continue hitting failed regions until TTL expires (and resolvers may ignore TTLs)
Regional fault isolation — a regional outage doesn't affect other regions Session continuity breaks when global LB redirects a user to a different region if session data is region-local
Capacity distribution across regions — traffic spikes in one geography don't overload a single data center Operational complexity multiplies significantly — multiple regional deployments, data replication, cross-region health monitoring
DDoS absorption at edge with anycast — attack traffic is spread across all PoPs Anycast requires BGP control and your own IP block — only feasible for large infrastructure operators; inaccessible to typical application teams
Enables data residency compliance — EU data stays in EU regions Cross-region data consistency is hard — eventual consistency is often the only practical model; strong consistency at global scale is extremely expensive

12 🎯 Interview Application

How do you use this concept in a design interview? What triggers it?

When an interviewer asks / says:

  • "How would you design this system for global users?" or "Users are distributed across the US, Europe, and Asia."
  • "What happens if an entire data center goes down?"
  • "How do you minimize latency for international users?"
  • "Walk me through your traffic routing architecture end to end."

What you say / do:

Introduce the two-tier model explicitly: "I'd use global load balancing to steer users to their nearest regional cluster, then local load balancing within each region to distribute across servers. For GSLB I'd use DNS-based latency routing with health-check-integrated failover for most services, or anycast for edge/CDN components where sub-second failover matters." Then proactively call out the TTL failover lag as a known limitation and address session locality if state is involved — showing you understand the second-order implications.

The trade-off statement (memorize this):

"If we use DNS-based global LB, we get simple geo-routing and regional failover with familiar tooling, but we pay with TTL-bounded failover lag — clients can continue hitting a dead region for up to our TTL window, and resolvers may cache even longer. For most stateless APIs this is acceptable; for real-time or financial systems I'd supplement with anycast or lower the TTL to 30 seconds, accepting higher DNS query volume as the cost."

13 ⚠️ Common Misconceptions & Gotchas

What do candidates get wrong? What nuance is the interviewer probing for?
❌
Misconception: DNS failover is near-instantaneous — update the record and clients redirect immediately.
✅
Reality: Clients cache DNS responses for the full TTL and many resolvers cache even longer in violation of the TTL spec. Real failover lag is often 2–5× the configured TTL. True instant failover requires anycast or an application-level redirect mechanism.
❌
Misconception: Global LB and CDN are the same thing.
✅
Reality: CDNs are one use case for global LB, but GSLB is broader. A GSLB routes to origin data centers; a CDN routes to edge caches. CDNs use GSLB internally, but a database or API service can use GSLB with no CDN layer at all.
❌
Misconception: Global LB handles session persistence — it always sends the same user to the same region.
✅
Reality: Global LB does not provide sticky sessions at the regional level (unless specifically configured). On failover, users will be redirected to a different region and their session data — if stored regionally — is unavailable. You must either replicate session state globally, use stateless JWTs, or accept session loss on regional failover.
❌
Misconception: Anycast is always better than DNS-based routing because failover is faster.
✅
Reality: Anycast routes to the topologically nearest PoP as seen by BGP, which may not be the lowest-latency PoP due to asymmetric routing. DNS-based latency routing actively measures RTT to each region and routes based on empirical latency data, which can outperform BGP topology for certain client locations. Also, anycast requires BGP control plane access most teams don't have.
