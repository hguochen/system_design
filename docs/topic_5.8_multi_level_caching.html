5.8 Multi-Level Caching
L1/L2, Local + Distributed

§0 🗺️ Topic Overview

Multi-level caching stacks two or more cache tiers — a fast in-process local cache (L1) and a larger shared distributed cache (L2) — to maximize hit rate while minimizing latency for the hottest data. The core tension is speed vs. coherence: L1 is nanosecond-fast but private per node, creating per-node stale windows; L2 is shared but adds a network round-trip. Mastering this topic means knowing when to add each layer, how to keep them coherent, and what failure modes emerge when both tiers go cold simultaneously.

  • 1. Layer responsibilities and sizing — L1 holds the hottest per-node keys; L2 holds the fleet-wide working set. Know how to size each tier given hit rate targets and object sizes.
  • 2. The read waterfall — L1 hit → return; L1 miss → L2 hit → fill L1; both miss → origin → fill L2 then L1. Be fluent with this path and the latency budget at each stage.
  • 3. Invalidation across all tiers — The hardest problem. A write must evict the key from L2 AND broadcast eviction to every node's L1. Know TTL-based and pub/sub-based strategies.
  • 4. Write path options — Write-invalidate (most common), write-through all tiers, or let TTL handle staleness. Know when each is appropriate.
  • 5. When NOT to add L1 — Without cache affinity (sticky routing), L1 hit rates are near zero. Adding L1 to a round-robin load balanced fleet adds complexity with no latency benefit.

§1 🎯 Goal of This Subtopic

After studying this, you should be able to design a complete multi-level cache architecture for a given system — specifying what lives at each tier, the TTL strategy per layer, the invalidation mechanism, and the consistency guarantees. You should also be able to identify when multi-level caching is unnecessary overhead versus a critical optimization.

§5 🧠 Core Definition

Multi-level caching is an architecture where data requests traverse a hierarchy of cache tiers — typically a fast in-process local cache (L1) followed by a slower but larger shared distributed cache (L2) — before reaching the origin data store, maximizing cache hit rate while minimizing read latency for the hottest data.

🔵 L1 — Local / In-Process ~0.1 µs

Lives inside the application process. No network hop. Examples: Caffeine (Java), lru-cache (Node), Python cachetools.

⚠️ Private to this node only — writes on other nodes don't update it.

🟠 L2 — Distributed / Shared ~1 ms

Shared across all application nodes over the network. Examples: Redis, Memcached, EVCache.

✅ A write to L2 is immediately visible to all nodes that query L2 next (for keys not in their L1).

§6 📦 Core Concepts

L1 Cache (Local / In-Process)

An in-memory cache living inside the application process itself — typically a bounded LRU or LFU map with TTL (e.g., Caffeine in Java, lru-cache in Node.js). Reads complete in under a microsecond because there is no network hop. The downside is privacy: L1 is visible only to the node it lives on. A write on node A does not automatically update L1 on nodes B, C, or D — creating a stale window bounded only by TTL or an explicit invalidation signal.

L2 Cache (Distributed / Shared)

A shared caching tier accessible by all application nodes over the network — most commonly Redis or Memcached. Because L2 is shared, a write to L2 is immediately visible to any node that queries L2 next (for keys not cached in their L1). L2 latency is typically 0.5–2ms in a co-located datacenter, which is fast for most workloads but still 10,000× slower than L1. L2 holds a much larger working set than any single node's L1, making it the primary hit driver for the fleet.

Cache Read Waterfall

The canonical read sequence: check L1 → on hit, return immediately; on miss, check L2 → on hit, populate L1 and return; on L2 miss, fetch from origin → populate L2 then L1, return. Hot data migrates up to L1 after the first L2 hit, making subsequent reads on that node L1-fast. The origin is only reached when both tiers miss simultaneously.

📊 Read Waterfall — All Three Scenarios

Three read scenarios: L1 hit is cheapest; both-miss reaches origin and risks thundering herd

L1 Invalidation

Because L1 is private per-node, invalidation is the hardest problem. Two primary strategies: (1) TTL-based expiry — set L1 TTL short (e.g., 5–10 seconds) so stale data naturally expires; simple but tolerates staleness proportional to TTL. (2) Pub/Sub broadcast — on write, publish an invalidation message to a Redis channel; all nodes subscribe and evict the key from their L1 immediately; bounds staleness to propagation delay (~milliseconds) but adds a message bus dependency.

Cache Affinity (Sticky Routing)

L1 hit rates are dramatically higher when the same key is consistently routed to the same application node — achieved via consistent hashing at the load balancer or session pinning. Without affinity in a round-robin fleet, each node independently cold-caches every key and L1 hit rates approach zero. With affinity, L1 hit rates for hot keys can reach 70–90%, reducing L2 and origin load by an order of magnitude.

§7 🔍 First Principles — Why Does This Exist?

A single-tier distributed cache (L2 only) solved the problem of offloading the database — but at extreme scale, even Redis at 1ms per read becomes the latency floor. If you have 10,000 RPS flowing through a single application node and 80% hit the same 1,000 hot keys, those 8,000 reads/second are paying 1ms each in network overhead — adding 8 full seconds of aggregate wait per second on that node alone.

The insight behind L1 is that network latency is unnecessary for data already in the same process. By keeping the hottest subset in-process, you short-circuit the network entirely for those reads. Only misses — which are rare for hot keys — pay the L2 round-trip. The compound effect is multiplicative: 90% L1 hit rate → 10× fewer L2 queries → 50% L2 hit rate on the remainder → 20× fewer origin queries compared to no caching. You can only achieve this through layering.

§8 🗺️ Mental Models

Model 1: The CPU Cache Hierarchy

Software multi-level caching mirrors the L1/L2/L3 cache inside a CPU. L1 CPU cache is tiny and blazing fast; L2 is larger and slightly slower; main memory (analogous to Redis/L2) is much larger but orders of magnitude slower; disk (analogous to the database) is authoritative but slowest. The engineering principle is identical: keep the hottest data in the fastest, closest tier and let misses fall through. Where it breaks down: CPU caches are hardware-coherent by design; software caches require explicit invalidation — the human must implement what the CPU does automatically.

Model 2: The Store's Shelf vs. Back Room vs. Warehouse

Think of L1 as the shelf stock visible to the cashier (fast, limited), L2 as the store's back room (larger, takes 30 seconds to retrieve), and the origin as the regional warehouse (authoritative, takes hours to ship). A popular product gets restocked from the back room to the shelf automatically. A product recall (invalidation) requires notifying every store's shelf simultaneously. Where it breaks down: in a real store, one cashier seeing the shelf means all cashiers see it. In multi-level caching, each application node has its own private shelf — the recall must reach every node independently.

Model 3: The Latency Budget Frame

Every cache read draws from a latency budget. L1 hit costs ~0.1µs; L2 hit costs ~1ms (10,000×); origin hit costs ~10ms (100,000×). A page load with 10 cache reads at 100% L2 costs 10ms in cache overhead alone. At 90% L1 hit rate: 9 × 0.1µs + 1 × 1ms ≈ 1ms total. L1 doesn't just improve average latency — it collapses the tail (p99 and p999) because tail latency is dominated by the slowest-tier reads. This model is your justification in interviews for adding L1 complexity.

§9 ⚙️ How It Works — Mechanics

✏️ Write-Invalidate Path (Most Common)

Write-invalidate: delete from L2, broadcast to all L1s, next read re-populates from fresh origin data

TTL Strategy

L1 TTL must always be ≤ L2 TTL. If L1 TTL > L2 TTL, a node could continue serving data from L1 even after L2's copy has expired — and when L1 finally expires, L2 is also gone, forcing an origin fetch that is older than intended. A good rule of thumb: L1 TTL = 5–30 seconds (short to bound stale window); L2 TTL = 5 minutes to several hours (depends on data freshness tolerance and write frequency).

Key Sizing

L1 should hold only the hottest key set. Size it as: (hot key count) × (avg object size). For example, 10,000 hot objects × 2KB each = 20MB per node — comfortably within JVM heap or process memory. L2 holds the full working set; size it based on total cacheable data volume × desired coverage, factoring in Redis cluster replication overhead.

⚠️ Mass invalidation thundering herd: After a deploy or cache flush, ALL nodes simultaneously miss both L1 and L2. Without protection, up to N application nodes each independently fetch the same key from origin — multiplying origin load by N. Mitigation: use a per-key mutex or Lua script at L2 (SET key lock NX EX 1) so only one node fetches and the rest wait, then reads the populated value.

§12 🎯 Interview Application

Trigger Phrases

  • "The system needs sub-millisecond latency for reads at high QPS"
  • "We have a hot key problem — certain keys are getting millions of hits per second on Redis"
  • "How would you further reduce latency? We already have Redis in place"
  • "The cache is becoming a bottleneck. What would you do?"

What You Say / Do

In the deep dive or optimization phase, introduce multi-level caching as a targeted optimization for hot keys. Frame it explicitly: "Given the read pattern here, I'd add an in-process L1 cache — something like Caffeine with a 10-second TTL — in front of Redis. The hot 1% of keys will stay L1-resident on each node, cutting L2 traffic by an order of magnitude. The key challenge is invalidation: I'd use a Redis Pub/Sub broadcast so all nodes evict their L1 entry within milliseconds of a write." Then address the thundering herd risk proactively to show depth.

💬 Trade-off statement (memorize this):

"If we add an L1 in-process cache, we get sub-millisecond reads for hot keys and dramatically reduce Redis traffic, but we accept a staleness window bounded by L1 TTL — or we add pub/sub invalidation to shrink that window to milliseconds at the cost of a message bus dependency. For this system, where user profile data changes infrequently and eventual consistency within 5 seconds is acceptable, L1 caching is the right call and TTL alone is sufficient."

§13 ⚠️ Common Misconceptions & Gotchas

❌
Misconception: Adding L1 in front of Redis always improves performance.
✅
Reality: L1 only provides significant benefit when the same node repeatedly serves the same keys (cache affinity). In a round-robin load balanced fleet where any node can serve any key, L1 hit rates are near zero and you're just adding memory overhead and invalidation complexity with no latency benefit.
❌
Misconception: L1 TTL can be longer than L2 TTL — L1 will just fall through to L2 on expiry anyway.
✅
Reality: If L1 TTL > L2 TTL, the L2 entry expires first. When L1 finally expires, L2 has a fresh version (or is empty). This means a node could serve a stale value from L1 for longer than the L2 copy would have been live — creating an inconsistency window that exceeds what you intended. L1 TTL must always be ≤ L2 TTL.
❌
Misconception: Invalidating L2 (deleting from Redis) is sufficient to keep the system consistent after a write.
✅
Reality: Invalidating L2 is necessary but not sufficient. All nodes' L1 caches may still hold the old value until their TTL expires or an explicit invalidation broadcast arrives. You must also invalidate all L1s (via pub/sub or wait for TTL), or you'll serve stale data for up to L1 TTL duration after every write.
❌
Misconception: Multi-level caching solves the thundering herd problem.
✅
Reality: Multi-level caching can make thundering herd worse at mass invalidation time. With N nodes each holding an L1 copy of a popular key, a mass invalidation (deploy, cache flush) causes ALL N nodes to simultaneously miss both L1 and L2 and fire concurrent origin requests for the same key. Without coalescing (mutex at L2), origin load spikes by N× compared to a single-tier setup.

1
Walk through the three read scenarios in a two-tier cache system: L1 hit, L1 miss / L2 hit, and both miss. What happens at each step and what is the approximate latency?
▼ Hint
💡 Think through: L1 hit → return (~0.1µs). L1 miss → query L2 → hit → populate L1 and return (~1ms). Both miss → fetch origin → populate L2 then L1 → return (~10ms). If you hesitate on populating order, revisit §9.
2
Your load balancer uses round-robin across 20 application nodes, each with an in-process L1 cache. Will L1 provide meaningful latency benefit? Why or why not?
▼ Hint
💡 No. Without cache affinity (sticky routing), each request for a given key lands on a different node each time. L1 hit rate approaches 1/20 = 5% — negligible. L1 helps only when the same node repeatedly serves the same key. Revisit §6 (Cache Affinity) and §13 if you were unsure.
3
A write occurs to a user's profile. You update the database and delete the key from Redis (L2). What could still go wrong, and what additional step is required?
▼ Hint
💡 All nodes' L1 caches may still hold the old version until their TTL expires. Reads from those nodes will return stale profile data. Additional step: broadcast an invalidation message (e.g., Redis Pub/Sub) to all nodes so they immediately evict the key from their local L1. Revisit §9 (Write Path) and §13 if you missed this.
4
Name a real production system that uses multi-level caching. Describe specifically how the two tiers are structured and why they chose to add the L1 tier.
▼ Hint
💡 Facebook Memcache (NSDI 2013): L2 = regional Memcached pools; L1 = per-web-server mcrouter local cache. L1 was added because at Facebook's scale, even low Memcached latency became a bottleneck when each page load required hundreds of cache reads. Revisit §10 for Netflix EVCache and Cloudflare alternatives.
5
A deploy flushes all caches — L1 and L2 go cold simultaneously across 50 nodes, handling 50,000 RPS. What specific failure mode is at highest risk, and how do you mitigate it?
▼ Hint
💡 Thundering herd amplified: all 50 nodes simultaneously miss both tiers and fire concurrent origin fetches for popular keys. At 50K RPS, this could generate thousands of simultaneous DB queries for the same 100 hot keys. Mitigation: (1) use a mutex at L2 (Redis SET NX) so only one node fetches each key; others wait and read from L2 after repopulation. (2) Warm caches during a canary deploy before full rollout. Revisit §9 callout warning.
