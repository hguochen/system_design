5.6 Cache Stampede & Thundering Herd

00 🗺️ Topic Overview

Cache stampede (also called thundering herd or dogpile effect) is a failure mode where a popular cached item expires and many concurrent requests simultaneously miss the cache, all stampeding to the origin database at once. The resulting spike in DB load can overwhelm the origin, cause cascading failures, and produce worse performance than having no cache at all.

📊 The Stampede Event — Timeline

Without mitigation: TTL expiry creates a synchronized spike — all concurrent requests miss the cache and hammer the DB simultaneously.

  • 1. Root cause mechanics — TTL expiration of a hot key is a synchronized event that every concurrent request observes simultaneously. This synchrony is the enemy.
  • 2. The five mitigations and their trade-offs — Probabilistic expiry, mutex/locking, background refresh, stale-while-revalidate, and TTL jitter each solve a different slice. Know when each applies.
  • 3. Request coalescing vs. probabilistic refresh — The two most interview-relevant mitigations. Be able to implement both conceptually and compare their consistency guarantees.
  • 4. Thundering herd on cold-start — Also occurs when a cache server restarts empty. Know this variant and its mitigation (pre-warming).
  • 5. Interaction with retry logic — Naive retries under load amplify thundering herd. Exponential backoff with jitter is the complement.

05 🧠 Core Definition

Cache stampede (thundering herd) is a failure mode where the simultaneous expiration or absence of a cached value causes a large number of concurrent requests to bypass the cache and hit the origin system at once, producing a load spike that can cascade into a system-wide outage. The problem is not the individual cache miss — it is the concurrent amplification of that miss across all in-flight requests.

06 📦 Core Concepts

Cache Miss Amplification

A single logical cache miss becomes many physical DB queries when request concurrency is high. For a key receiving 10,000 req/s with a 50ms recompute window, up to 500 requests can arrive between the first miss and the repopulated cache. Each sees an empty cache and fires an independent DB query — 500 simultaneous origin fetches for one expired key.

Thundering Herd vs. Hot Key Problem

Related but distinct. A hot key problem is sustained overload on a single DB key due to high read volume. A thundering herd is a transient spike triggered by a synchronized miss event (TTL expiry, server restart, deployment). Hot key mitigation (replication, local caching) helps; stampede prevention helps — but they are not the same fix.

Probabilistic Early Expiration (XFetch)

Before TTL reaches zero, each request independently computes a probability of refreshing early based on how close to expiry the key is and how expensive the recompute is. The probability increases exponentially as TTL approaches zero. No coordination needed — randomness distributes refreshes across time, preventing synchronized expiry.

Mutex-Based Refresh (Request Coalescing)

When a cache miss is detected, the first request acquires a distributed lock (SETNX lock_key "1" EX <ttl>) and fetches from DB. All other concurrent misses either wait and re-check, or are served stale data. Guarantees at-most-one DB fetch per key per refresh window. Critical: the lock must have a TTL.

Stale-While-Revalidate

An HTTP/CDN caching directive (RFC 5861) that instructs the cache to serve stale content immediately while triggering an asynchronous background refresh. The client gets a fast response; the DB gets a single background refresh request. CDN-layer equivalent of mutex-based refresh without application code changes.

TTL Jitter

Instead of all keys sharing a fixed TTL, each key's TTL is randomized: ttl = base_ttl + random(-jitter, +jitter). This spreads expirations over time, preventing batch-loaded keys from creating a synchronized stampede wave. Zero read-path overhead.

07 🔍 First Principles — Why Does This Exist?

Caches work by trading freshness for performance: instead of computing or fetching the same answer repeatedly, you store it once and serve it many times. The performance gain depends on the cache hit rate staying high. The moment a hot cached value expires, the cache provides zero protection — but all the requests that were being absorbed by the cache are now in-flight simultaneously when expiry occurs.

The DB, designed to handle only residual traffic that the cache doesn't cover, suddenly receives the full uncached load all at once. This is a systems-level thundering herd: many processes synchronized by a single shared event (the TTL clock), all waking up and competing for the same resource at the same moment. The paradox: the more successful your cache, the worse the stampede when it fails — because the DB has been starved of load and hasn't been sized to handle direct traffic.

08 🗺️ Mental Models

Model 1: The Midnight Door Rush

Imagine a nightclub that closes its doors at midnight (TTL expiry). Everyone waiting outside rushes in simultaneously the moment the doors open — instead of a steady trickle. The stampede is caused not by the volume of people but by the synchronization of their arrival. Jitter is the equivalent of opening the doors at random times between 11:55 PM and 12:05 PM — the crowd self-distributes.

Where it breaks down: This doesn't capture the lock-holder failure scenario — if the first person in (lock holder) collapses in the doorway, you need a bouncer (lock TTL + stale fallback) to handle it.

Model 2: The Checked-Out Document

When a popular shared document is checked out for editing (cache miss + lock acquired), everyone else who needs it pauses and waits. This is correct — you don't want 500 people making the same edit simultaneously. But if the person who checked it out gets fired mid-edit (lock holder crashes), the document stays locked forever unless there's a policy to reclaim it after N minutes (lock TTL).

Where it breaks down: In the office analogy, waiting is visible. In distributed systems, "waiting" means holding an HTTP connection open, consuming threads — the cost of waiting is non-obvious until connections exhaust.

Model 3: Probabilistic Early Boarding

A train departs at 10:00. Passengers who know it sometimes leaves early board randomly between 9:45 and 9:59. This is XFetch: each passenger independently decides when to "board early" based on how close to departure time it is and how long it takes to get to the platform (recompute cost). Randomness breaks synchrony without central coordination.

Where it breaks down: If the cost of early departure (recompute) is very high, passengers avoid it and you still get a rush at 10:00. XFetch's beta parameter controls this sensitivity.

🔒 Mutex / Request Coalescing — State Machine

Mutex pattern: winner fetches from DB; losers wait or serve stale. The lock TTL is a safety must-have.

📈 XFetch — Probabilistic Early Refresh Curve

As TTL approaches zero, XFetch probability of early refresh approaches 1 — requests self-distribute the refresh load without coordination.

Cold-Start Thundering Herd: When a cache server restarts empty (after crash, deployment, or scale-out), 100% of requests miss until warm. Mitigation: pre-warm by replaying hot keys before routing live traffic, restore from Redis RDB/AOF, or use circuit breakers to shed load during the warmup window.

11 ⚖️ Trade-offs

✅ Benefit❌ Cost / Limitation
XFetch: zero coordination, self-distributing Serves slightly stale data more often; beta tuning requires empirical calibration
Mutex: at-most-one DB fetch, waiting clients get fresh data Lock holder failure blocks refreshes; waiting adds latency; requires distributed lock infra
Stale-while-revalidate: zero added client latency, single background fetch Not suitable for strongly consistent data (balances, inventory); stale window may be unacceptable
TTL jitter: trivial to implement, zero read-path overhead Only helps for batch-loaded keys; doesn't prevent a single viral hot key from stampeding
Background refresh: read path never sees a miss for warmed keys Requires separate process; wasted resources on unpopular keys; misses unexpected viral content

13 ⚠️ Common Misconceptions & Gotchas

❌Misconception: Cache stampede only happens when the cache server is down.
✅Reality: The most common cause is normal TTL expiration of a hot key. The cache is working perfectly — the stampede is caused by the expiry event itself, not a cache failure.
❌Misconception: Setting a longer TTL prevents stampedes.
✅Reality: A longer TTL delays the stampede but doesn't prevent it. When the key eventually expires, the same thundering herd occurs. Jitter, coalescing, or probabilistic expiry are the correct fixes.
❌Misconception: A mutex lock around cache miss handling is sufficient on its own.
✅Reality: A mutex without a TTL is a time bomb. If the lock holder crashes between acquiring the lock and writing to cache, the lock is never released and all future requests for that key are permanently serialized or blocked. Always set a lock expiry.
❌Misconception: Stale-while-revalidate causes unacceptable consistency problems.
✅Reality: Staleness is acceptable in many real systems — social feed rankings, product listings, news headlines. The key question is the staleness window tolerance, not whether any staleness exists.
