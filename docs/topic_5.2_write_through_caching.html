5.2 Write-Through Caching

00 🗺️ Topic Overview

What This Topic Is About

Write-through caching is a write strategy where every data write is synchronously propagated to both the cache and the backing database before the client is acknowledged. It is the eager counterpart to cache-aside's lazy loading — while cache-aside populates the cache only on read misses, write-through ensures the cache is always in sync with the DB immediately after any write. The central tension here is consistency vs. write latency: you pay two I/Os per write in exchange for zero stale reads after writes.

🎯 What to Focus On

  • 1. The write-through sequence and ACK timingThe client waits until both the cache write AND the database write complete. Know this cold — it's the defining property that separates write-through from every other strategy.
  • 2. The consistency guarantee — and its exact scopeWrite-through guarantees freshness for keys that have been written through. It does NOT guarantee a fully warm cache — cold nodes and evicted keys still miss. Know what it does and does not promise.
  • 3. Write amplification as the key costEvery write becomes two writes. For write-heavy workloads this doubles your I/O burden. The decision rule is simple: write-through is for read-heavy, consistency-critical systems; avoid it for write-heavy ones.
  • 4. Failure atomicity — the partial write problemCache write + DB write is not an atomic operation. If the DB write fails after the cache write succeeds, you have a stale cache entry. Know the mitigation: write DB first, invalidate cache on DB failure.
  • 5. Contrast with cache-aside and write-backWrite-through sits between cache-aside (lazy, stale-prone) and write-back (deferred, durability risk). Be able to place all three on the consistency-performance spectrum and select the right one for a given workload.

05 🧠 Core Definition

Write-through caching is a write strategy where every data write is synchronously written to both the cache and the backing database before the operation is acknowledged to the client, ensuring the cache is always in sync with the source of truth.

06 📦 Core Concepts

Synchronous Write-Through Path

Every write request updates the cache first, then the database (or both in parallel), and the client receives acknowledgment only after both writes complete. This two-phase write is what gives write-through its consistency guarantee. The cost is latency: the client waits for two I/O operations instead of one.

Cache Consistency Guarantee

Because every write populates the cache with the latest value, any subsequent read will hit the cache and return fresh data. Unlike cache-aside (where the cache is populated lazily on read misses), write-through has no window of stale reads after a write. This is the defining property and the primary reason to choose it.

Write Amplification

Write-through doubles the number of write operations — every logical write becomes two physical writes. For write-heavy workloads (high write/read ratio), this amplification becomes a bottleneck. A system doing 100k writes/sec with write-through is pushing 200k write operations to two different storage tiers.

Cold-Start / Cache Warming Gap

Write-through only populates cache entries on writes. If you deploy a new cache node, restart the cache, or have data that was last written before the cache existed, those keys will be absent. Reads will hit the database until a write for that key occurs. For read-heavy workloads, this means cache hit rate starts at 0 and climbs only as writes happen — you often pair write-through with a cache pre-warming step on deployment.

Failure Atomicity Problem

Write-through creates a two-write sequence (cache → DB) that is not atomic. If the DB write fails after the cache write succeeds, the cache holds data that the DB does not — a consistency violation in reverse. Good implementations handle this by writing to the DB first and only updating the cache on DB success, then invalidating the cache key on any DB failure.

07 🔍 First Principles — Why Does This Exist?

The root problem is cache inconsistency after writes. In a naive caching setup (cache-aside), you write to the database directly and the cache is only populated when a read misses. This means that immediately after a write, the cache may still hold the old value — any reader that hits the cache before the next miss+reload will see stale data. For systems like shopping carts, banking balances, and user auth state, stale reads are a correctness problem, not just a performance nuisance.

Write-through was the solution: make the cache an always-up-to-date mirror of the database by routing all writes through it. This trades write latency for read correctness. The pattern originates from CPU cache design — L1/L2 write-through caches existed for exactly the same reason: ensuring that memory-mapped reads always reflect the latest written value.

08 🗺️ Mental Models

Model 1: The Synchronized Mirror

Think of write-through as keeping a mirror perfectly synchronized with the original. Every time the original changes, the mirror is updated immediately — before anyone looks at it. The cost is that every update requires two actions instead of one. The benefit is that anyone consulting the mirror always sees the current state. The model breaks down when you ask "what if the mirror update fails?" — physical mirrors don't fail, but your write-through cache can, which is why the failure atomicity problem matters.

Model 2: The Bank Teller and Ledger Book

Imagine a bank teller (cache) keeping their own notes alongside the official ledger (database). In write-through, every transaction is recorded in both the teller's notes and the ledger before the customer leaves. The customer waits longer than if the teller just updated their own notes and deferred the ledger, but there's never a discrepancy between teller's notes and the official record. On failure: if the teller updates their notes but the ledger fails, the teller must erase their note — this is the rollback/invalidation pattern.

Model 3: The Write Amplification Tax

Every write-through operation charges a "tax" — one extra write to the cache on top of the database write. For read-heavy systems (95% reads), the tax is nearly invisible. For write-heavy systems (50%+ writes), the tax dominates: you're doubling your write I/O for marginal read benefit. This model lets you quickly eliminate write-through as a candidate for write-heavy workloads without reasoning through the full trade-off every time.

09 ⚙️ How It Works — Mechanics

Normal (Happy) Path

  1. Client sends a write request (e.g., SET user:123 → {name: "Gary"})
  2. Application writes the value to the cache: cache.set("user:123", data)
  3. Immediately after, application writes the same value to the database
  4. Only after both writes succeed does the app return success to the client
  5. Subsequent reads for user:123 hit the cache — no DB hit required

Write Order — Cache First vs. DB First

Cache first: Write cache → write DB. Risk: if DB write fails, cache holds a value the DB doesn't. Must invalidate cache key on DB failure.

DB first: Write DB → write cache. Risk: if cache write fails after DB success, the next read misses cache and hits DB — acceptable degradation but not a correctness violation. DB-first is generally safer.

Failure Handling

If the DB write fails: invalidate (delete) the cache key so subsequent reads go to the DB and get the pre-write state. Do NOT leave the cache holding the new value the DB rejected. If the cache write fails (DB-first approach): log the failure, proceed — the next read will simply hit DB and miss cache. Correctness is maintained.

Key Formula

Write latency = t_cache_write + t_db_write (serial) or max(t_cache_write, t_db_write) (parallel). For a 1ms cache write and 5ms DB write: serial write-through costs 6ms vs 5ms direct DB — a 20% overhead that scales linearly with write volume.

13 ⚠️ Common Misconceptions & Gotchas

❌Write-through means the cache is always fully populated — if you write-through, you never have cache misses.
✅Write-through only populates keys that have been written. New cache deployments, cache evictions, or data never written-through will still miss. Write-through only guarantees freshness for keys that were written through.
❌Write-through is always better than cache-aside because it's "more consistent."
✅Write-through has double write latency and write amplification. For write-heavy workloads it's strictly worse than cache-aside. Consistency matters only if you have stale-read problems — many systems tolerate eventual consistency fine and don't need write-through.
❌Write-through is atomic — cache and DB are always consistent.
✅There are two writes, not one atomic operation. If the DB write fails after the cache write succeeds, you have a consistency violation. Good implementations write DB first and invalidate the cache on failure. True atomicity requires distributed transactions — expensive.
❌Write-through and read-through are the same strategy.
✅Write-through is about the write path — ensuring the cache is updated on every write. Read-through is about the read path — the cache itself fetches from the DB on a miss. They are complementary, not synonymous, and can be combined.
