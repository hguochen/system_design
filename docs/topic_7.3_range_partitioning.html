7.3 Range Partitioning
Contiguous Key Ranges, Splits, and the Hot-Shard Problem

§0 🗺️ Topic Overview

Range partitioning splits a dataset across shards by dividing the key space into contiguous, ordered ranges — shard 1 holds keys A–F, shard 2 holds G–M, and so on. Unlike hash partitioning, which scatters keys pseudo-randomly to spread load evenly, range partitioning preserves the natural ordering of keys, so a range query like "all events between 9am and 10am" touches only the shards that overlap that range instead of fanning out to the whole cluster. The core tension is query efficiency vs. load distribution: preserving order unlocks efficient range scans, but the same property makes range partitioning acutely vulnerable to hot shards whenever the write pattern is monotonic.

  • 1. Why range partitioning exists at all — Hash partitioning destroys order; if the access pattern depends on scanning contiguous ranges, hashing forces a scatter-gather across every shard. Range partitioning trades load-balance guarantees for query efficiency.
  • 2. The hot-shard failure mode — The single most-tested scenario: a monotonically increasing key (timestamp, auto-increment ID) sends every new write to the same shard. Know this cold — it's the signature gotcha.
  • 3. Split point selection and rebalancing — Shards aren't fixed forever; as a shard grows, it splits at the MEDIAN key of stored data, not the key-space midpoint.
  • 4. Range vs. hash partitioning — the direct comparison — Range wins when range queries and ordered scans dominate; hash wins when point lookups dominate and even load distribution is the priority.
  • 5. Real systems that use it — Bigtable/HBase, MongoDB ranged sharding, and CockroachDB/Spanner are the canonical examples; know how each handles the hot-shard problem in practice.

§5 🧠 Core Definition

Range partitioning is a sharding strategy that assigns each shard a contiguous, ordered range of the key space, so that data with nearby keys is co-located on the same shard — enabling efficient range scans at the cost of uneven load distribution when writes are concentrated in a narrow part of the key range.

🔵 Range Partitioning Ordered

Shards own contiguous key ranges. Range scans and ordered iteration touch only 1–2 shards. Examples: Bigtable, HBase, MongoDB ranged sharding, CockroachDB.

⚠️ Monotonic keys create severe hot-shard risk.

🟠 Hash Partitioning Scattered

Keys hashed to shards, spreading load evenly regardless of write pattern. Examples: DynamoDB (partition key), Cassandra, most consistent-hashing systems.

✅ No hot-shard risk from monotonic keys, but range queries require scatter-gather.

§6 📦 Core Concepts

Key Range / Shard Boundary

Each shard owns a contiguous half-open interval of the key space, e.g., [A, F) meaning keys starting with A through E. A routing component (a config service, coordinator, or client-side metadata cache) maintains the boundary map and directs each read/write to the shard whose range contains the key. Bigtable calls this the METADATA table; CockroachDB calls it the range descriptor.

📊 Shard Boundary Map & Range Scan

A range query only touches the shards whose boundaries overlap the requested range

Range Scan

A query that reads all keys within a contiguous interval, e.g., "all orders between 2026-01-01 and 2026-01-31." Because range partitioning preserves key order, a range scan touches only the shards whose ranges overlap the query interval — often just one or two shards — instead of fanning out to the entire cluster as hash partitioning would require.

Shard Splitting

As a shard accumulates data and crosses a size threshold (commonly 1–4GB in systems like Bigtable), it is split into two shards at a chosen split point. The split point should be the MEDIAN KEY of the actual stored data (not the midpoint of the theoretical key space) so both resulting shards end up roughly equal in size, since real-world key distributions are rarely uniform.

Monotonic Key / Hot Shard

A monotonic key (auto-increment ID, timestamp, sequence number) always increases, so every new write's key falls after all previously written keys — meaning every new write lands on the same (rightmost) shard. This concentrates 100% of write throughput onto one shard while every other shard is idle for writes — the defining failure mode of range partitioning.

Key Salting / Prefixing

A mitigation for monotonic keys: prepend a low-cardinality, well-distributed prefix (e.g., a hash of the user ID mod N) to the natural key before partitioning. Writes now spread across N buckets/shards. Range scans across the entire dataset now require querying all N buckets, but scans within a bucket (e.g., one user's timeline) remain efficient — the trade-off is scoped rather than eliminated.

§7 🔍 First Principles — Why Does This Exist?

Once a dataset outgrows a single machine, it must be split across nodes — but the splitting strategy has consequences for query patterns. Hash partitioning was the first instinct for many systems because it distributes load evenly and requires no coordination on boundaries: hash the key, mod by shard count, done. But hashing intentionally destroys the ordering relationship between keys — key "2026-01-01" and key "2026-01-02" end up on completely unrelated shards. For any workload that needs to read a contiguous range of data, hash partitioning forces a scatter-gather: query every shard and merge results client-side, even though the caller only wanted a narrow slice of data.

Range partitioning was invented to preserve locality of reference for ordered data. By keeping shard assignment aligned with the natural order of the key space, a range query becomes a targeted operation against a small number of shards instead of a full-cluster fan-out. The cost of this choice is that the even-load guarantee hash partitioning offered for free is now something you must engineer explicitly — auto-splitting, load-aware rebalancing, and key design (salting monotonic keys) all exist specifically to claw back the load-balance properties that ordering sacrifices.

🔥 The Hot-Shard Problem — Monotonic Key Write Pattern

Every new write lands on the shard covering the highest key range — the signature range-partitioning gotcha

§8 🗺️ Mental Models

Model 1: The Library Shelving Analogy

Range partitioning is like a library that shelves books alphabetically by author surname across multiple rooms: Room 1 holds A–F, Room 2 holds G–M, and so on. If you want "everything by an author whose name starts with C," you walk straight to Room 1 — you don't search every room. But if a wildly popular author whose name starts with "Z" gets released every week, Room 6 becomes overcrowded while Room 1 sits quiet. Where it breaks down: a library can't easily "split Room 1 into 1a and 1b" mid-operation the way a range-partitioned database dynamically re-splits a shard.

Model 2: The Sorted Array vs. Hash Table Frame

Think of the entire cluster as one giant sorted array (range partitioning) versus one giant hash table (hash partitioning). A sorted array gives O(log n + k) range queries but degrades badly under skewed insertions at one end — like repeatedly appending to an array. A hash table gives O(1) average point lookups and naturally distributes inserts, but has no concept of "next key." This model works well for reasoning about Big-O trade-offs quickly; it breaks down because a distributed shard isn't literally a single sorted array — physical shard boundaries and network hops matter in ways an in-memory array doesn't capture.

Model 3: The Highway On-Ramp Bottleneck

Imagine a highway divided into ordered mile-marker sections, each maintained by a different regional crew (shard). If all new traffic (writes) always enters at mile marker 100 — think a timestamp-keyed log — then the crew responsible for mile marker 100 is perpetually overwhelmed while crews at mile markers 0–99 have nothing new to do. The fix mirrors key salting: build multiple on-ramps (buckets) that feed into the same corridor at staggered points, spreading incoming traffic across several crews instead of funneling it into one.

§9 ⚙️ How It Works — Mechanics

Read/write path (happy case): A client or coordinator computes the operation's key, then a routing layer consults the boundary/metadata map — an ordered list of (range_start, range_end, shard_id) tuples — via binary search to find the owning shard. A point lookup is a single-shard operation; a range query [K1, K2] identifies all shards whose range overlaps and queries only those, merging results in key order.

✂️ Shard Splitting — Median Key vs. Midpoint

Splitting at the alphabetic midpoint ignores real data distribution; median-by-volume keeps both halves balanced

TTL / Split Threshold

Each shard tracks its data size (or row count) against a configured threshold (e.g., 1–4GB per Bigtable tablet). When exceeded, the split point is chosen as the MEDIAN KEY among the shard's actual stored data. The metadata/boundary map is updated atomically to reflect the new two-range assignment. In-flight requests during a split are typically handled via a brief redirect/retry: the router detects a stale boundary map (via a version number or "not the owner" response) and re-fetches the updated map.

Handling the Monotonic-Key Case

If the partition key is monotonically increasing, every new write's key sorts after all existing keys, so it always lands on the shard owning the highest range. Mitigations: salting (prepend an N-way distributed prefix before partitioning — range scans over a single user's data remain efficient, scoped to one bucket); reversal (store a bit-reversed or inverted timestamp, scattering insert order while sacrificing natural chronological ordering); and load-aware auto-splitting (split not just on size but on request-rate/QPS to a shard, proactively splitting a hot range even if it isn't large).

Google Bigtable Rows are range-partitioned into "tablets," auto-split at a size threshold and reassigned across tablet servers The canonical range-partitioning system; row-key design (e.g., reversing a monotonic timestamp) is explicitly recommended in Google's docs to avoid hot tablets
HBase Directly modeled on Bigtable — regions are range-partitioned by row key and auto-split by size Region hotspotting from sequential row keys (timestamps) is a well-documented HBase anti-pattern with the same salting fix

What You Say / Do

In the data model / partitioning section of the design, introduce range partitioning when the access pattern includes ordered scans or range queries. Say something like: "Since we need to query events by time window, I'd range-partition by timestamp so a query for a given hour only touches one or two shards. The risk is that a raw timestamp key is monotonic, so I'd salt it with a low-cardinality prefix — hash of device ID mod N — to spread writes across N shards while keeping per-device time-range scans efficient within each bucket."

💬 Trade-off statement (memorize this):

"If we range-partition by timestamp, we get efficient range scans for time-window queries, but we risk concentrating all writes on the newest shard since timestamps are monotonic. For this system, I'd salt the key with a device or shard-bucket prefix to spread writes across N shards, accepting that a full cross-bucket chronological scan now requires merging N ranges instead of reading one."

§13 ⚠️ Common Misconceptions & Gotchas

❌
Misconception: Range partitioning always gives worse load distribution than hash partitioning, so it should be avoided.
✅
Reality: Range partitioning gives worse load distribution only when the write pattern is skewed toward one part of the key space (most commonly, monotonic keys). For uniformly distributed key spaces with heavy range-query needs, range partitioning is strictly better than hash partitioning — you get both reasonable load balance AND efficient range scans.
❌
Misconception: Splitting a hot shard in half will always fix a hot-shard problem.
✅
Reality: If the hot shard's hotness comes from a monotonically increasing key, splitting it in half only delays the problem — the new "highest" half-shard immediately becomes the new hot shard as writes continue to increase. Splitting solves size imbalance, not write-pattern-driven hot shards; that requires salting or key redesign.
❌
Misconception: The split point for a shard should be the midpoint of its key range (e.g., splitting "A–Z" at "M").
✅
Reality: The split point should be the median key by actual stored data volume. If 90% of your data starts with "A–C," splitting "A–Z" at "M" leaves one shard nearly empty and the other still overloaded — you must split based on where the data actually is, not where the alphabet happens to sit.
❌
Misconception: Once you salt a monotonic key, you lose the ability to do range queries entirely.
✅
Reality: You lose global ordering across the full key space, but range queries scoped to a single salt bucket (e.g., one user's or one device's timeline) remain efficient. Cross-bucket range queries require querying all N buckets and merging, which is more expensive but still far cheaper than the pre-partitioning full scan.

What is range partitioning, and how does it differ from hash partitioning in how it maps keys to shards?

You split a shard covering key range "A–Z" exactly at "M" because that seemed like the natural midpoint. Two weeks later, the "A–M" shard is still overloaded while "N–Z" is nearly empty. What went wrong, and how should the split have been performed?
