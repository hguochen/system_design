6.4 Geo-routing and Anycast
Steering users to the nearest edge

§0 🗺️ Topic Overview

Geo-routing and anycast are the two primary mechanisms CDNs use to route each user to the geographically or topologically nearest PoP. Anycast works at the network layer: the same IP prefix is announced from every PoP via BGP, and the internet's routing infrastructure naturally delivers packets to the closest announcer. DNS geo-routing works at the application layer: the authoritative DNS server returns a different A record per region based on the resolver's inferred location. Together, they reduce round-trip time from 150–300ms (cross-continent) to 5–30ms (nearest PoP). The core tension is routing speed vs. accuracy: anycast is automatic and instant but topologically imprecise; DNS geo-routing is flexible and accurate but requires ECS support and tolerates TTL-bounded staleness during failover.

  • 1. Anycast BGP mechanics — The same /24 prefix is announced from all PoPs. BGP selects the path with fewest AS hops. Know that "nearest" is topological, not geographic — a well-peered far PoP can beat a poorly-peered close one.
  • 2. DNS geo-routing path — Trace the full resolution: client → recursive resolver → authoritative DNS with geo-logic → PoP A record. The authoritative DNS sees the resolver's IP, not the client's. This is the root cause of geo-routing inaccuracy.
  • 3. EDNS Client Subnet (ECS) — Partial fix: resolver forwards a /24 client prefix to authoritative DNS. Improves accuracy at the cost of privacy. Not universally supported.
  • 4. Failover behavior — Anycast: BGP route withdrawal → re-convergence in 30–120s (or <5s with BFD). DNS geo-routing: TTL expiry before failover takes effect — short TTL = faster failover but higher DNS load.
  • 5. When to use each — Anycast for UDP protocols (DNS itself, QUIC), DDoS absorption, and zero-overhead routing. DNS geo-routing for fine-grained per-user policies and TCP traffic with complex routing rules.

§5 🧠 Core Definition

Geo-routing and anycast are complementary traffic-steering mechanisms that route each user to the CDN PoP with the lowest round-trip time: anycast achieves this at the network layer by advertising one IP from all PoPs and letting BGP converge packets to the nearest announcer; DNS geo-routing achieves it at the application layer by resolving a domain to the A record of the nearest PoP based on the client's inferred location.

🔵 Anycast Layer 3

  • Same IP prefix announced from all PoPs via BGP
  • Routing at the packet level — zero DNS overhead
  • Ideal for UDP (DNS, QUIC) and DDoS scrubbing
  • Failover: automatic BGP route withdrawal

🟠 DNS Geo-routing Layer 7

  • Different A records returned per geographic region
  • Routing at resolution time — TTL-cached per client
  • Ideal for fine-grained per-user routing policies
  • Failover: bounded by DNS TTL (set 30–60s)

§6 📦 Core Concepts

Anycast IP Routing

A network addressing scheme where the same IP prefix (e.g., 192.0.2.0/24) is announced from multiple autonomous systems (CDN PoPs) via BGP simultaneously. When a packet is addressed to that IP, every internet router along the path forwards it toward whichever announcer has the best BGP path — typically the one with the fewest AS hops. No per-request DNS is needed; routing is a property of the IP-layer infrastructure. Cloudflare's 300+ PoP network operates on a small number of anycast prefixes — every Cloudflare customer shares the same IP range globally.

DNS-Based Geo-routing

The authoritative DNS server for a CDN domain inspects the source IP of each DNS query and returns a different A record based on the inferred geography of that resolver. A resolver in Singapore receives the Singapore PoP IP; a resolver in Frankfurt gets the Frankfurt PoP IP. The routing decision is made once per TTL window at DNS resolution time — not per packet. AWS Route 53 offers this as "latency-based routing" (routes to lowest-latency region rather than nearest region by geo) and "geolocation routing" (routes by country/continent).

EDNS Client Subnet (ECS)

An extension to DNS (RFC 7871) where the recursive resolver includes a truncated version of the client's IP address (e.g., the /24 prefix) in the DNS query sent to the authoritative server. Without ECS, a user in Tokyo using Google's 8.8.8.8 resolver causes the authoritative DNS to route based on Mountain View — a continent away. With ECS, the resolver says "this query is on behalf of a client in 203.0.113.0/24", and the authoritative server routes accurately. Privacy cost: the client /24 is exposed to the authoritative DNS operator and potentially logged.

BGP Route Withdrawal and Re-convergence

When an anycast PoP goes offline, its BGP sessions drop and its route announcements are withdrawn. Neighboring routers detect the missing announcements and propagate the withdrawal through the BGP mesh, removing the failed PoP from routing tables globally. Traffic re-routes to the next-best PoP as BGP reconverges — typically 30–120 seconds without optimization. With Bidirectional Forwarding Detection (BFD), failure is detected in milliseconds and convergence completes in 1–5 seconds. This automatic failover requires no operator action and no DNS change.

TTL and DNS Failover

DNS geo-routing failover is bounded by TTL. When a PoP goes unhealthy, the authoritative DNS updates its response — but clients who already resolved the old IP continue using it until their cached TTL expires. A 300-second TTL means up to 5 minutes of clients hitting a failed PoP. A 30-second TTL reduces this to 30 seconds, but 10× the DNS query volume hits the authoritative server. Most CDNs use 60–120 seconds as the practical default, with the ability to drop to 30 seconds during incident response.

§7 🔍 First Principles — Why Does This Exist?

The root constraint is physics: the speed of light in fiber is approximately 200,000 km/s. A round trip between Sydney and London (17,000 km of cable) takes at minimum 170ms — before any processing. For a web page requiring 10 sequential round trips, that's 1.7 seconds of pure propagation delay. No amount of faster servers or better code eliminates this — only proximity does.

CDNs solve this by placing servers at hundreds of PoPs worldwide. But proximity is only valuable if users are actually routed to nearby PoPs. Before geo-routing existed, DNS returned a single IP for a domain — all traffic hit the origin server, regardless of whether a closer PoP existed 5ms away. The user in Tokyo paid 200ms to reach a US server even if a Tokyo PoP sat idle.

Anycast and DNS geo-routing solve the routing half of the problem. They transform a single-IP address space into a globally distributed routing fabric where each user is automatically directed to their nearest point of service — without the user, application, or client knowing anything about the underlying topology.

⏱️ Why Proximity Matters — RTT Without vs. With Geo-routing

Geo-routing collapses RTT from 200ms → 10ms for cache hits; origin is only reached on cache miss

§8 🗺️ Mental Models

Model 1: The Post Office Network (Anycast)

Imagine a postal service where thousands of sorting facilities worldwide all carry the same address label. When you drop a letter in a mailbox, the postal routing system sends it to whichever facility is closest — not because your letter has routing instructions, but because the routing infrastructure has pre-configured "send mail of this type to the nearest facility." Anycast works identically: the IP prefix is the "class of address," and BGP routing tables are the pre-configured rules that forward packets to the nearest announcer. Where it breaks down: BGP "nearest" means fewest AS hops, not fewest kilometers — a geographically distant PoP with better peering can win over a physically closer one with worse BGP metrics.

Model 2: The Smart Receptionist (DNS Geo-routing)

Think of the authoritative DNS server as a multinational company's central receptionist. When you call in, the receptionist asks "where are you calling from?" based on your caller ID (the resolver's IP), then transfers you to the nearest regional office (PoP IP). ECS is like telling the telephone exchange to include your local area code in the transfer request — so the receptionist routes you based on your actual location, not the exchange's. Where it breaks down: once transferred, you stay on that line for the duration of the TTL — even if the regional office goes down, you're stuck until the TTL expires and you call the receptionist again.

Model 3: Two Layers — City and Building

In production CDNs, anycast and DNS geo-routing work together at different granularities. Anycast decides which city (datacenter) receives the packet at the IP layer. DNS geo-routing decides which building (cache cluster or origin shield) within that city handles the request at the application layer. This two-layer model is why Cloudflare uses anycast globally while AWS CloudFront uses DNS-based latency routing — different architectures for the same end goal, or sometimes combined in the same system.

§11 ⚖️ Trade-offs

✅ Benefit❌ Cost / Limitation
Anycast: automatic failover — BGP route withdrawal re-routes traffic without operator action or DNS changes BGP convergence is slow — 30–120s without BFD; clients hit the failed PoP during convergence; BFD helps but requires peering support
Anycast: zero routing overhead — no per-request DNS lookup for routing; routing is baked into the IP layer BGP "nearest" ≠ geographic nearest — topological proximity (AS hops) can route users to a farther PoP if the closer one has worse peering relationships
DNS geo-routing: fine-grained policies — can route by country, city, ISP, weight, or real-time latency measurement (Route 53 latency-based) DNS sees resolver IP, not user IP — without ECS, a user's resolver location can cause routing to a distant PoP; ECS partial fix with privacy cost
DNS geo-routing: fast failover possible — short TTL (30s) + health checks enables failover in under one minute Short TTL increases DNS load — every client must re-query when TTL expires; 10× TTL reduction = 10× DNS query volume on authoritative servers
"If we choose anycast, we get automatic BGP-driven failover and zero routing overhead, but we pay with 30–120 second convergence windows and imprecise geographic targeting from BGP topology. For DDoS scrubbing and UDP protocols, anycast is the right call; for fine-grained per-user routing policies and TCP HTTPS traffic, DNS geo-routing gives us more control."

§13 ⚠️ Common Misconceptions & Gotchas

❌Misconception: "Anycast routes to the geographically nearest PoP."
✅Reality: Anycast routes to the topologically nearest announcer as determined by BGP path selection — based on AS hop count, local preference, and MED, not geographic distance. A PoP with a shorter AS path (better peering) can win over a physically closer PoP with worse BGP metrics.
❌Misconception: "DNS geo-routing sees the user's actual IP address."
✅Reality: The authoritative DNS server sees the recursive resolver's IP address, not the end user's. A user in Tokyo using Google's 8.8.8.8 resolver causes the authoritative DNS to route based on Mountain View. ECS partially addresses this — but requires resolver support and exposes the client /24 to the authoritative server.
❌Misconception: "Short DNS TTL means instant failover."
✅Reality: TTL determines when a cached entry expires. Clients that already resolved the old IP continue using it until their TTL expires. Failover latency = max(remaining TTL on cached entry). Also, some resolvers and OS caches ignore TTL minimums and cache longer — so real failover can exceed the configured TTL.
❌Misconception: "Anycast and DNS geo-routing are alternatives — you pick one."
✅Reality: Production CDNs commonly use both in a layered model — anycast for L3/L4 routing (which PoP gets the packet) and DNS for L7 routing (which cache tier or origin shield to use within a region). They solve routing at different layers, not the same problem.
