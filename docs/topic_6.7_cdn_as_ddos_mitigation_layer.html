6.7 CDN as DDoS Mitigation Layer
Absorb, scrub, and filter attacks at the edge

§0 🗺️ Topic Overview

A CDN is not only a latency cache — its globally distributed, high-capacity Anycast edge makes it one of the most effective DDoS mitigation layers available. The same properties that deliver content fast (hundreds of PoPs, Anycast routing, terabits of aggregate capacity, edge caching) also let it absorb and filter attack traffic before it reaches your origin. Mastering this topic means knowing why the edge is the right place to stop an attack, which layers of attack a CDN can and cannot stop, and the single most common way engineers accidentally leave a CDN-protected origin wide open.

  • 1. The three attack classes and where the CDN intercepts each — volumetric (L3/4 bandwidth floods), protocol/state-exhaustion (SYN floods, reflection), and application-layer (L7 HTTP floods). Know which the edge absorbs trivially and which need real intelligence.
  • 2. Anycast as the absorption mechanism — the same IP announced from every PoP means a global attack is diluted across the whole network instead of converging on one box. Explain it in one breath.
  • 3. Origin hiding — and the bypass that defeats it — the CDN only helps if attackers can't reach the origin directly. The #1 failure is origin IP leakage; the fix is firewalling the origin to allow only CDN IP ranges.
  • 4. Caching and rate limiting as offload — cached content never touches origin, absorbing floods for free. Attackers counter with cache-busting; you counter with query normalization, rate limits, WAF, and challenges.
  • 5. What the CDN does NOT solve — non-proxied protocols, a leaked origin, false positives against flash crowds, and TLS-termination trust. Know the limits.

§5 🧠 Core Definition

Using a CDN as a DDoS mitigation layer means placing a globally distributed, high-capacity Anycast edge network in front of your origin so that attack traffic is absorbed, filtered, and rate-limited at the edge — diluted across hundreds of PoPs and stopped before it can saturate the origin's bandwidth, exhaust its connection state, or overload its application tier.

🛡️ Without a CDN exposed

Origin's single uplink (~10–100 Gbps) and finite connection table face the full attack directly.

⚠️ A multi-Tbps flood saturates the pipe instantly — nothing you provision at one site can win.

🌍 With a CDN edge absorbed

Anycast spreads the attack across hundreds of PoPs; edge scrubs L3/4 and filters L7 before origin.

✅ Origin only sees clean, rate-limited, cache-offloaded traffic — if it's locked to CDN IPs.

§6 📦 Core Concepts

Anycast Absorption

Anycast announces the same IP address from every PoP, and BGP delivers each packet to the topologically nearest PoP. For DDoS this is transformative: a globe-spanning botnet has its traffic automatically split across the whole edge rather than converging on one location. A 3 Tbps attack hitting 300 PoPs is ~10 Gbps per PoP — easily absorbed — whereas the same attack aimed at a single origin uplink is instantly fatal. Anycast turns "attacker vs. one server" into "attacker vs. the CDN's entire aggregate capacity."

🌐 Anycast Dilution — Same Attack, Two Targets

The identical attack that saturates one origin uplink is diluted to a trivial per-PoP load across an Anycast edge

Origin Hiding (and Origin Bypass)

When a CDN fronts your service, DNS resolves to the CDN's edge IPs, so attackers see only the edge. The origin's real IP is meant to be secret. The catastrophic failure mode is origin bypass: if the real IP leaks — through historical DNS records, SSL/TLS certificate transparency logs, unprotected subdomains (mail., ftp., staging.), email/SMTP headers, or a service that connects out and reveals its address — an attacker hits the origin directly and routes around all CDN protection. The fix is a firewall allowlist that accepts traffic only from the CDN's published IP ranges, plus authenticated origin pull (mTLS or a secret header) so even allowlisted spoofers are rejected.

The Attack-Layer Taxonomy (L3/4 vs L7)

Volumetric (L3/4) floods aim to saturate raw bandwidth (UDP floods, reflection/amplification off DNS/NTP/memcached). Protocol/state-exhaustion (L3/4) attacks like SYN floods fill finite connection or firewall-state tables using little bandwidth. Application-layer (L7) attacks send requests that look legitimate (HTTP GET/POST floods, Slowloris slow-reads) to exhaust CPU, worker threads, or the database. The higher the layer, the harder to distinguish attack from real load — which is why L7 needs behavioral intelligence, not just capacity.

📊 Attack Taxonomy — Target, Unit, CDN Mechanism

Higher layers are harder to filter: capacity beats volumetric, but L7 needs behavioral intelligence

Caching as an Absorption Layer

Every request satisfied from the edge cache is a request the origin never sees. For attacks targeting cacheable content (static pages, images, assets), the CDN absorbs the flood essentially for free — the edge just keeps serving from cache at PoP capacity. Attackers respond with cache-busting: appending random query strings so every request is a cache MISS that forwards to origin. Defenses are to normalize the cache key (ignore unknown params) or rate-limit requests that miss cache.

Edge Filtering: WAF, Rate Limiting, and Challenges

For traffic that must reach dynamic endpoints, the CDN applies intelligence at the edge: a Web Application Firewall matches known-bad signatures and anomalies; rate limiting caps requests per IP / token / path; bot management scores clients by fingerprint and behavior; and challenges (JS challenge, CAPTCHA, or proof-of-work) force clients to prove they're a real browser. Legitimate users pass transparently or with minor friction; cheap, high-volume bots fail cheaply — and all of it runs in the edge PoP, close to the attacker and far from the origin.

§7 🔍 First Principles — Why Does This Exist?

The root problem is an asymmetry of scale and geography. A single origin data center has a fixed, relatively small uplink (often 10–100 Gbps) and finite state resources — connection tables, file descriptors, worker threads, DB connections. An attacker commanding a botnet of hundreds of thousands of devices (or abusing open reflectors) can trivially generate multiple Tbps and billions of pps from all over the world. No matter how much you over-provision one location, the attacker can provision more cheaply, because attack capacity is stolen and yours is paid for. You cannot win a capacity race at one point in space.

The CDN inverts the asymmetry. By operating hundreds of PoPs with enormous aggregate capacity and announcing your service via Anycast, it forces the attacker to fight the entire network instead of one server — and the geographic spread means BGP automatically splits the attack toward many PoPs, each seeing only a fraction. On top of raw absorption, the edge is the ideal place to filter: it's close to the attacker (scrub before traffic crosses expensive long-haul links), it has global visibility (a pattern seen in one PoP can be blocked network-wide), and it serves cached content without ever bothering origin. Standalone "scrubbing centers" existed before CDNs — folding mitigation into the always-on content edge made it continuous, latency-free in the common case, and economically viable.

⚖️ The Core Asymmetry — Why One Location Can't Win

You can't out-buy stolen botnet capacity at one location — you must distribute across an edge network

§8 🗺️ Mental Models

Model 1: Many Shock Absorbers vs. One Windshield

Picture the attack as a truckload of gravel dumped at speed. Aimed at a single windshield (your origin), it shatters instantly. Spread across hundreds of shock absorbers (Anycast PoPs), each takes a handful of pebbles and shrugs it off. The CDN's whole value is distributing the impact so no single surface bears the full force. Where it breaks down: for L7 attacks the problem isn't force but disguise — a few "pebbles" that look exactly like raindrops.

Model 2: The Hidden Castle Behind a Public Gatehouse

The origin is a castle; the CDN edge is a ring of gatehouses the public interacts with. All roads lead to the gatehouses, which inspect everyone (WAF, rate limits, challenges) and forward only vetted, minimal traffic to the castle via a secret, guarded tunnel (authenticated origin pull). The fatal mistake is leaving a map to the castle lying around (leaked origin IP) or a back door unlocked (origin firewall not allowlisting the CDN). Where it breaks down: unlike a physical castle, the origin's location can leak through dozens of subtle side channels.

Model 3: The Bouncer Who Costs the Attacker More Than You

Challenges reframe the fight economically: force every client to do a small amount of work to enter. A real browser does it invisibly; a botnet trying to send millions of rps must now pay CPU per request, collapsing its throughput. You're not blocking every packet — you're making the attack uneconomical. Where it breaks down: sophisticated bots using real browsers / residential proxies pass challenges, and the friction causes false positives for legitimate users too.

§9 ⚙️ How It Works — Mechanics

Normal path (no attack)

DNS resolves to the CDN's Anycast IP; BGP routes packets to the nearest PoP; the edge terminates TLS, checks cache, serves on hit or fetches from origin over a secured (often shielded) path on miss. The origin only ever sees cache-miss, authenticated traffic from CDN IPs.

Under an L3/4 volumetric attack (e.g., DNS reflection flood)

Spoofed-source flood packets are routed by Anycast to many PoPs, diluting the volume geographically. Each PoP's edge scrubbing drops traffic that doesn't belong — packets to closed ports, malformed packets, known reflection signatures, spoofed sources failing sanity checks. Legitimate traffic continues; the attack never crosses to origin because the origin firewall only accepts CDN IPs anyway.

Under an L3/4 protocol attack (SYN flood)

A flood of SYNs aims to exhaust connection-table state. The edge answers with SYN cookies — encoding connection state into the returned sequence number instead of allocating a table entry — so half-open connections cost the edge nothing. Only clients that complete the handshake get a real connection and are proxied onward.

Under an L7 application attack (HTTP flood / cache-busting)

Requests arrive that are individually well-formed. Cache offload handles cacheable content first — served at the edge, never reaching origin. For cache-busting or dynamic endpoints, the edge applies rate limiting, WAF rules, and bot scoring, then issues a challenge (JS → CAPTCHA → managed challenge / proof-of-work) to suspicious clients. Only the small residue of vetted requests is forwarded to origin, ideally through an origin shield (a single upstream PoP consolidating fetches — the same multi-level caching idea from §5.8 / §6.6).

⚠️ The critical prerequisite: the origin must be locked so it cannot be reached except through the CDN — firewall allowlist of CDN IP ranges + authenticated origin pull (mTLS or secret header). Otherwise every mechanism above is bypassable by hitting the origin IP directly.

Key parameters worth remembering

Reflection amplification factors — memcached ~×50,000, NTP ~×550, DNS ~×28–54 — explain why a small attacker generates a huge flood, and why absorbing at the edge matters. Rate-limit thresholds are tuned per endpoint (login endpoints far stricter than a public homepage) to balance protection against false positives.

Trigger Phrases

  • "How do you keep the origin from being overwhelmed?"

What You Say / Do

Raise it during the non-functional-requirements or "harden the design" phase, right after establishing the CDN for content delivery. Say: "Since we're already fronting cacheable content with a CDN, I'd lean on that same edge as the DDoS mitigation layer. Anycast spreads volumetric attacks across the whole PoP network so no single location is saturated; the edge scrubs L3/4 floods and applies WAF, rate limiting, and challenges for L7. Critically, I'd lock the origin firewall to accept traffic only from the CDN's IP ranges and use authenticated origin pull — otherwise a leaked origin IP lets attackers bypass the CDN entirely. For non-HTTP protocols I'd add an L3/4 product like Magic Transit / Shield Advanced / Prolexic."

💬 Trade-off statement (memorize this):

"Fronting the origin with a CDN's Anycast edge gives us multi-Tbps absorption plus L7 filtering that no single data center could match, at the cost of TLS-termination trust in the provider and the discipline of keeping the origin unreachable except through the CDN. For a public web service where availability under attack matters more than avoiding a third-party dependency, that's the right trade — provided we lock down the origin, or the whole thing is bypassable."

§13 ⚠️ Common Misconceptions & Gotchas

❌
Misconception: Putting a CDN in front of your service automatically makes it DDoS-proof.
✅
Reality: Only traffic that actually goes through the CDN is protected. If the origin's real IP is exposed (old DNS records, SSL cert transparency logs, unprotected subdomains, email headers), attackers hit it directly and bypass the CDN entirely. You must firewall the origin to accept only CDN IP ranges and use authenticated origin pull.
❌
Misconception: A CDN stops all types of DDoS the same way.
✅
Reality: Volumetric L3/4 attacks are absorbed by Anycast capacity and edge scrubbing; protocol attacks like SYN floods use SYN cookies; but L7 application floods look like legitimate traffic and require behavioral intelligence — WAF, rate limiting, bot scoring, and challenges. Capacity alone doesn't stop a well-crafted L7 attack.
❌
Misconception: Caching means the origin is safe from any HTTP flood.
✅
Reality: Caching only offloads requests for cacheable content. Attackers use cache-busting — appending random query strings so every request is a MISS that forwards to origin. You must normalize/ignore unknown query params in the cache key or rate-limit cache-miss traffic, or the flood passes straight through.
❌
Misconception: DDoS mitigation at the edge is free of downsides.
✅
Reality: The CDN terminates TLS (it sees your plaintext), becomes a third-party dependency and potential single point of failure, and its L7 heuristics can false-positive on legitimate users — challenging or blocking real customers during a flash crowd. These are real costs to weigh, not zero.
