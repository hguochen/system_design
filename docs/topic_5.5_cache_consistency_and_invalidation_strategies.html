5.5 Cache Consistency and Invalidation Strategies

00 🗺️ Topic Overview

What This Topic Is About

Cache consistency is the problem of keeping a cache synchronized with the source of truth (usually a database) when data changes. The central tension is freshness vs. simplicity: the more strictly you enforce consistency, the more complex your invalidation logic becomes and the lower your effective cache hit rate. Mastering this subtopic means knowing which invalidation strategy fits which write pattern, and understanding the race conditions that make "just delete the key on write" harder than it sounds.

🎯 What to Focus On

  • 1. The four invalidation approaches and when each appliesTTL-based expiry, write-invalidation (delete on write), write-through (update on write), and CDC-based propagation are not interchangeable. Each has a different consistency window, infrastructure requirement, and failure mode. Know the trade-off matrix cold.
  • 2. The cache-aside race conditionThe classic delete-on-write pattern has a subtle race: a read and a write can interleave so that a stale value ends up cached. This is the most common "gotcha" interviewers probe for. Know the ABA sequence of events and the mitigation (TTL safety net, version tags, or Lua atomic scripts).
  • 3. TTL is the floor, not the ceilingTTL bounds your worst-case staleness window. It is not a substitute for proactive invalidation. Production systems use TTL + explicit invalidation together — TTL is the safety net; invalidation is the fast path.
  • 4. Surrogate keys / tag-based invalidation for CDNCDN edge caches use HTTP purge APIs and surrogate key tags. Know how tag-based invalidation lets you invalidate thousands of edges with a single API call keyed by content tag, not by URL.
  • 5. Consistency models for cachesCaches provide strong consistency (write-through), bounded staleness (TTL), or eventual consistency (lazy expiry). Matching the model to the business requirement is the skill — financial data needs strong or short TTL; social media likes can tolerate 30-second staleness.

05 🧠 Core Definition

What is it, in one sentence?

Cache invalidation is the set of strategies for removing or updating cached entries when the underlying data changes, with the goal of preventing stale reads while maintaining cache effectiveness. The core challenge is that caches are optimistic read accelerators while writes are the enemy of that optimism — every write to the source of truth creates a window where the cache may hold incorrect data.

06 📦 Core Concepts

The essential building blocks — terms and ideas you must have solid before going deeper.

TTL-Based Expiry

TTL (Time To Live) stamps each cache entry with a maximum age. Once elapsed, the entry is treated as a miss — the next read fetches fresh data from the database and re-populates the cache. TTL is the simplest invalidation mechanism: it requires zero coordination between the writer and the cache, and it bounds the worst-case staleness to the TTL duration. The costs: stale data may be served during the TTL window, and when many entries expire simultaneously you risk a cache stampede. TTL is the foundation — every serious caching setup uses TTL as a safety net regardless of which other strategy is also in place.

Write-Invalidation (Delete on Write)

When a write occurs to the database, the application also deletes the corresponding cache key. The next read for that key misses, fetches fresh data from the database, and populates a new cache entry. This is the most common pattern with cache-aside and is conceptually simple. The danger is the ABA race condition: if a concurrent read fetches old data from the DB just before the write completes, and then writes that old value to the cache just after the invalidation delete, the cache is repopulated with stale data. TTL is essential as a safety net to bound this race window.

Write-Through

Write-through keeps the cache and database synchronized by updating both on every write. A write to the database is immediately followed by an update (not just a delete) of the corresponding cache entry. This eliminates the stale window at the cost of increased write latency and cache pollution — you're caching data that may never be read again. Write-through is most appropriate when read frequency is very high, the working set is well-defined, and strong consistency is required.

CDC (Change Data Capture) Invalidation

CDC reads the database's transaction log (PostgreSQL WAL, MySQL binlog) via a connector like Debezium, publishes change events to a message queue (Kafka), and a cache invalidation consumer deletes or updates the affected cache keys. This fully decouples the application write path from cache management — the application doesn't need to know a cache exists. CDC provides reliable, ordered invalidation with a slight lag (50–500ms typically). It enables fan-out: one DB change can propagate to multiple independent cache layers. The trade-off is additional infrastructure and operational complexity.

Surrogate Key / Tag-Based Invalidation

Used primarily in CDN and HTTP reverse proxy caches (Varnish, Fastly, Cloudflare). When a response is generated, the origin adds a header tagging it with logical content identifiers: Surrogate-Key: article:123 author:456. CDN edge nodes index each cached URL against its tags. When that content changes, the origin sends one API call: PURGE article:123. The CDN finds all cached responses bearing that tag across all edge nodes globally and evicts them. This scales to thousands of edge invalidations from a single event without URL enumeration.

Stale-While-Revalidate

A hybrid pattern where the cache serves the existing stale entry immediately while triggering an asynchronous background refresh to fetch and store the current value. The client gets sub-millisecond response (stale data), and the cache self-heals without a blocking cache miss. This is the semantics of HTTP Cache-Control: stale-while-revalidate and is used in CDN edges and application-level caches. It is appropriate for content where slight staleness is tolerable; unacceptable for inventory counts, financial balances, or anything requiring strong consistency.

07 🔍 First Principles — Why Does This Exist?

What fundamental problem does this concept solve? Why was it invented?

Caches exist because reads vastly outnumber writes in most systems, and the database is a bottleneck for reads. The entire value proposition of a cache is to serve reads from fast memory without hitting the database. But this only works if the cached data is correct — which requires the cache to be notified or to expire whenever the database changes.

The fundamental tension: the write path and the read path are decoupled by design, but consistency requires they be coordinated. There is no free lunch: the more strictly you enforce consistency, the more coordination overhead you add to the write path, the lower your cache efficiency, and the harder your system is to operate at scale.

Before TTL and explicit invalidation were formalized, early systems either served stale data indefinitely (cache never invalidated) or bypassed the cache on every write (defeating its purpose). Facebook's 2013 memcache paper is a landmark case study — at millions of operations per second, even the simple "delete on write" approach required a lease mechanism to prevent the race condition from causing correctness violations at scale. The progression from TTL → write-invalidation → CDC reflects the industry's increasing write throughput over time.

08 🗺️ Mental Models

Intuition frames that help you reason about this concept fast — especially under interview pressure.

Model 1: The Newspaper Delivery

Think of the cache as a stack of yesterday's newspapers. TTL is like setting a policy: "throw out any paper older than 24 hours." Write-invalidation is like having the delivery person knock and remove the old paper the moment a new edition is printed. Write-through means the presses hand-deliver an updated copy to your door with every new edition. CDC is a notification dispatch: the presses send a signal to central dispatch, which tells all delivery people to replace their papers.

Where this breaks down: The newspaper analogy doesn't capture the race condition — the delivery person can knock right as you're reading the old paper, and if there's a delay, you might put the old paper back after they removed it. This is exactly the ABA race.

Model 2: The ABA Race as a Timeline

Draw a timeline with two threads. Thread A (Reader): [miss → read DB: gets V1 → …pause…→ write V1 to cache]. Thread B (Writer): [write V2 to DB → delete cache key]. The race happens when B's delete lands between A's DB read and A's cache write. A doesn't know V2 was written — it merrily writes V1 back to cache after the key was already cleaned up. The TTL safety net catches this: V1 will expire within TTL seconds even if it's written incorrectly.

Model 3: The Consistency Dial

Imagine a dial with three positions: Strong (write-through — always consistent, highest write cost), Bounded Staleness (TTL + write-invalidation — small consistency window, medium cost), and Eventual (TTL-only — widest window, lowest cost). The correct setting is determined by business requirement, not technical preference: financial transactions → Strong, product listings → Bounded Staleness (~30s), social engagement counts → Eventual (60s is fine). Most systems run different cache layers at different dial settings for different data types — you rarely pick one setting for everything.

09 ⚙️ How It Works — Mechanics

Step-by-step explanation of the internal mechanism.

TTL Expiry — Lazy + Active

Each entry is written with a creation timestamp and a TTL (seconds). On read, the cache computes now - created_at. If it exceeds TTL, the entry is a miss even though the key exists in memory (lazy expiry). A background sweep thread periodically scans for expired keys and frees their memory (active expiry). Both mechanisms run in parallel: lazy expiry ensures correctness on access; active expiry prevents unbounded memory growth from expired keys never re-read.

Write-Invalidation — Normal Flow and Race

Happy path: App writes to DB → app calls cache.delete(key) → next cache read misses → DB read → cache repopulated with V2.

ABA race sequence:

  1. Thread A: cache miss → reads from DB → gets value V1
  2. Thread B: writes V2 to DB → calls cache.delete(key)
  3. Thread A: writes V1 to cache (stale — after B's delete!)
  4. Cache now holds V1, DB holds V2

Mitigations: Short TTL as expiry safety net; version/timestamp check before cache write (reject if cached version is newer); Facebook lease tokens (cache issues a read lease on miss; a concurrent write revokes the lease, preventing the stale write-back); Redis Lua scripts for atomic check-then-set.

Write-Through — Mechanics and Failure Modes

App writes to DB and cache in sequence (or within a transaction wrapper). Cache is always up-to-date after the write path completes. Failure mode: if cache write fails after DB write succeeds, DB is ahead of cache. The safe response is to delete the cache key (forcing the next read to re-fetch fresh data) rather than leaving the stale entry. Belt-and-suspenders: always add a TTL even to write-through entries — if a cache write failure goes undetected, TTL bounds the staleness window.

CDC Pipeline — Mechanics

  1. App writes to DB (no cache knowledge required in app code)
  2. DB appends write to transaction log (WAL / binlog)
  3. CDC connector (Debezium) reads the log, publishes change event to Kafka topic
  4. Cache invalidation consumer group reads from Kafka
  5. Consumer calls cache.delete(key) or cache.set(key, new_value)

The consistency window = steps 1→5 latency, typically 50–500ms. Multiple consumers on the same Kafka topic enable fan-out: one DB write can invalidate Redis, the CDN, and an application-level cache simultaneously. Kafka's consumer group model ensures exactly-once processing per partition.

Surrogate Key CDN Invalidation — Mechanics

On response generation, the origin adds: Surrogate-Key: article:123 author:456 tag:news. The CDN edge stores a mapping of each cached URL to all its tags. On a content update event, origin sends: PURGE article:123. CDN finds all cached URLs tagged with article:123 across all edge PoPs and evicts them. Global propagation time: ~150ms on Cloudflare, ~200ms on Fastly. This makes tag-based invalidation O(1) from the origin's perspective regardless of how many URLs share the tag — critical for CMS systems where one template change invalidates thousands of pages.

11 ⚖️ Trade-offs

Every design decision has a cost. What are you giving up?
✅ Benefit❌ Cost / Limitation
TTL: Zero coordination on write path; simple to implement; works with any caching strategy Stale data served during TTL window; short TTL → high miss rate + stampede risk; not a substitute for explicit invalidation
Write-invalidation: Fast path to freshness on write; easy to implement; works with cache-aside ABA race condition allows stale data back into cache; app must coordinate cache + DB writes; brittle under high concurrency without additional guards
Write-through: Strong consistency — cache and DB always in sync after write completes Higher write latency (two writes per operation); cache pollution from unread keys; cache write failure leaves system in inconsistent state
CDC: Decoupled from app; reliable, ordered, replay-able; enables multi-cache fan-out with one event Extra infrastructure (Debezium, Kafka, consumer service); 50–500ms propagation lag; significant operational complexity vs. simpler patterns
Surrogate keys: O(1) bulk CDN invalidation regardless of URL count; works at global edge scale CDN-vendor-specific; requires origin to tag responses at generation time; 100–500ms propagation latency; no help for application-layer in-process caches

12 🎯 Interview Application

How do you use this concept in a design interview? What triggers it?

When an interviewer asks / says:

  • "How do you keep the cache consistent when data changes?"
  • "What happens if a user updates their profile but sees the old data for several minutes?"
  • "Your system has high write throughput — how do you invalidate the cache?"
  • "Design a caching layer for a product page that must show accurate inventory counts"

What you say / do:

In the cache design or deep dive section, proactively state your invalidation strategy and tie it explicitly to the consistency requirement. Always call out the write-invalidation ABA race condition to show depth — it signals to interviewers that you know the real complexity. Then propose the mitigation (TTL + version tag, or CDC for high-write systems). Classify the business requirement first (strong / bounded / eventual) before recommending a strategy.

"If we choose write-invalidation, we get fast consistency with low infrastructure overhead, but we pay the risk of the ABA race condition: a concurrent read can repopulate the cache with stale data after the delete. For this system, write-invalidation with a 30-second TTL safety net is the right call because write frequency is low and bounded staleness is acceptable. If we needed sub-second consistency — for example, inventory counts during a flash sale — I'd layer in explicit key deletion triggered by the inventory service, or switch to write-through for that specific data type."

13 ⚠️ Common Misconceptions & Gotchas

What do candidates get wrong? What nuance is the interviewer probing for?
❌
Misconception: Just delete the cache key on write — that handles consistency.
✅
Reality: Delete-on-write has the ABA race condition: a concurrent read can fetch stale data and write it back to cache after the delete has already fired. Without a TTL or version check as a safety net, this stale value persists indefinitely. The mitigation is short TTL + version checks, or Facebook's lease mechanism at scale.
❌
Misconception: Write-through provides perfect consistency — the cache is always correct.
✅
Reality: Write-through has its own failure mode: the DB write succeeds but the cache update fails, leaving the cache behind the DB. The correct response is to delete the cache key on cache-write failure (forcing re-fetch on next read). Write-through also pollutes the cache with cold keys — data written but never read — wasting cache capacity. TTL is still necessary even with write-through.
❌
Misconception: TTL-based invalidation is too slow/weak — serious systems don't rely on it.
✅
Reality: TTL is a foundational mechanism present in every major production cache. The right TTL is workload-dependent: 1 second for hot inventory, 5 minutes for product descriptions, 24 hours for static assets. TTL is not weak — it's precise, predictable, and always necessary as a safety net even when explicit invalidation is also in place.
❌
Misconception: CDC-based invalidation is only for massive companies with huge engineering teams.
✅
Reality: Debezium + Kafka is operationally accessible and is the right choice whenever the write path should not be coupled to cache management, write throughput is high, or multiple cache layers need coordinated invalidation. CDC is appropriate at mid-scale — not just at Facebook/Netflix scale. The managed CDC tools have lowered the barrier significantly.
