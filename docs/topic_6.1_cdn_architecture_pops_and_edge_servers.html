6.1 CDN Architecture — PoPs & Edge Servers

00 🗺️ Topic Overview

A CDN (Content Delivery Network) is a globally distributed network of edge servers — organized into Points of Presence (PoPs) — that cache and serve content from locations physically close to end users. The core tension it resolves: you want a single source of truth (origin), but you need to serve that content to users around the world with sub-30ms latency. Without a CDN, a user in Singapore accessing a US-hosted asset pays 200–300ms in round-trip time — every single request.

🌍 CDN Global Architecture — User → PoP → Origin Shield → Origin

CDN topology: Users route to nearest PoP via DNS/anycast. Cache HITs return immediately (~10ms). Cache MISSes fan in through origin shield to protect origin from N×traffic amplification.

  • 1. PoP topology and what lives at the edge — A PoP is not just a cache server. It contains load balancers, TLS terminators, caching layers, and sometimes compute (edge functions). Know what each component does and why co-locating them at the edge reduces latency.
  • 2. How a user request reaches the nearest PoP — DNS-based geo-routing and anycast are the two main mechanisms. Be able to explain each and why anycast is preferred for global scale.
  • 3. Cache hit vs. cache miss path — The performance contract only holds on a cache hit. On a miss, the edge must fetch from origin (cache fill). Know the full round-trip for both paths and the latency implications.
  • 4. Origin shield (mid-tier caching) — Between edge PoPs and origin, many CDNs add a shield layer to concentrate cache-miss traffic and protect origin from thundering herd at edge scale.
  • 5. When CDN helps vs. hurts — CDN is not a universal accelerator. For non-cacheable content (personalized, no-store), CDN adds latency. Knowing the limits separates solid answers from great ones.

01 🎯 Goal

Be able to describe the full CDN architecture — from DNS resolution routing a user to the nearest PoP, through the edge server's cache lookup, to a cache miss that fills from origin via origin shield — and explain why each component exists. 05 🧠 Core Definition

A CDN (Content Delivery Network) is a globally distributed network of edge servers — organized into Points of Presence (PoPs) — that cache and serve content from locations physically close to end users, reducing latency by eliminating long round trips to a centralized origin server.

A PoP is the atomic unit: a co-located cluster of load balancers, TLS terminators, and caching servers deployed in Internet Exchange Points (IXPs) and carrier-neutral datacenters worldwide. When a user requests content, DNS or anycast routing directs them to the nearest PoP, which serves the response from its local cache — bypassing the origin entirely on a hit.

06 📦 Core Concepts

🏢 Inside a PoP — Component Architecture

Inside a PoP: TLS terminates locally (saving 150–200ms vs. origin handshake), cache tiers serve hits in microseconds–milliseconds. Origin is only contacted on a full cache miss through both L1 and L2.

Point of Presence (PoP)

A PoP is a CDN's edge datacenter — a physical location where the CDN operates servers close to end users. A typical PoP includes: an Anycast IP receiver or DNS resolution handler, one or more load balancers, TLS termination infrastructure (so TLS handshakes complete locally), a multi-layer cache (L1 in-memory + L2 SSD), and increasingly edge compute (Cloudflare Workers, Lambda@Edge). PoPs are strategically placed at Internet Exchange Points (IXPs) to peer directly with ISPs, minimizing network hops to users.

Edge Server

An edge server is an individual server within a PoP that handles user requests. It checks its local cache for the requested content; on a hit, it responds immediately. On a miss, it fetches from origin (or origin shield) and caches the response for subsequent requests. Edge servers are stateless with respect to business logic — their only job is cache lookup and cache fill.

Origin Shield (Mid-Tier Cache)

An origin shield is an optional but critical intermediate caching layer between edge PoPs and the origin server. When hundreds of PoPs simultaneously miss the same object (after a cache invalidation or popularity spike), they would each independently fetch from origin — creating N×traffic amplification. The shield acts as a single rendezvous point: all PoPs funnel cache-miss requests through it, so the origin sees at most one fetch per object regardless of how many PoPs request it. This is essential for protecting origin at CDN scale.

Cache Fill and Cache Hit Ratio

Cache fill is the process of fetching content from origin to populate the edge cache after a miss. The cache hit ratio (CHR) is the fraction of requests served from cache without hitting origin. CHR is the primary CDN effectiveness metric — a CHR of 95% means only 5% of requests travel to origin. CHR is driven by content TTL, content uniqueness, and request volume per object.

Routing to the Nearest PoP

DNS-based geo-routing maps the requesting DNS resolver's IP to a geographically appropriate PoP — simple, but inaccurate when resolver IPs don't reflect user location. Anycast routing assigns the same IP to all PoPs and relies on BGP to route packets to the topologically closest PoP — more accurate, lower latency, no DNS TTL dependency. Anycast is preferred by latency-critical CDNs (Cloudflare uses it extensively).

07 🔍 First Principles — Why Does This Exist?

The speed of light is finite. A round trip from Singapore to a US East Coast server takes 200–300ms at the physical limit — and real network routing adds more. For web content, this means every asset paid this latency tax on every request.

Early internet content was hosted in single datacenters because storage was expensive; geographic distribution was cost-prohibitive. As bandwidth costs fell and static content (images, video, scripts) became the dominant volume of internet traffic, the latency problem became critical. Users in Tokyo shouldn't pay the latency cost of a server in Virginia for a static JavaScript file that hasn't changed in weeks.

The solution is obvious in retrospect: cache the content close to where users are. CDNs were born to do exactly this — Akamai's founding insight in 1998 was that you could dramatically improve web performance by distributed caching at the network edge, without requiring website owners to operate their own global infrastructure.

08 🗺️ Mental Models

Model 1: The Library Branch Network 📚

Think of the CDN as a public library system. The main library (origin) has every book (all content). Branch libraries (PoPs) hold popular books locally — anyone nearby can check them out instantly. If a branch doesn't have the book, they order it from the main library and keep a copy for future requests. The origin shield is a regional distribution center: branch libraries request from the regional hub, not directly from the main library, so the main library doesn't get a thousand simultaneous orders for the same new bestseller. Where it breaks: library books don't expire — CDN TTLs mean your cached copy may go stale and be discarded even if the original hasn't changed.

Model 2: Cache Hit Ratio Is Your CDN's Report Card 📊

The CDN is only as good as its cache hit ratio. Every miss is a tax — you paid CDN fees, added a hop, and still made an origin round trip. A CDN with 40% CHR on personalized API responses is not improving performance; it's adding infrastructure complexity with no latency benefit and potential latency harm. Before adding CDN to a design, estimate the CHR: how cacheable is the content, what's the TTL, how many requests will warm the cache per object?

Model 3: Anycast as "Nearest Subway Station" 🚇

Anycast routing is like a city where every subway station has the same address — you enter "subway" into your navigation and it routes you to the nearest one. BGP does the routing automatically based on network topology. If a PoP goes down, BGP reconverges and automatically routes traffic to the next-nearest PoP — no manual failover needed. Where it breaks: BGP convergence isn't instant (seconds to minutes), so there's a brief disruption window during PoP failure.

09 ⚙️ How It Works — Mechanics

⚡ Cache HIT vs. Cache MISS — Full Request Flow Comparison

Key insight: the HIT path completely bypasses origin (the expensive round-trip). The MISS path pays full origin latency plus the CDN overhead. Your system's performance is a CHR-weighted average of these two paths.

✅ CACHE HIT — ~15ms

  1. User's browser DNS query resolves to nearest PoP IP (anycast or DNS geo-routing)
  2. TLS handshake completes at PoP (~10ms local vs. ~150ms to origin)
  3. Edge server checks L1 (RAM) then L2 (SSD) cache → object found, not expired
  4. Edge returns response with Age and X-Cache: HIT headers
  5. Total: ~10–30ms depending on user-PoP distance

❌ CACHE MISS — ~150–300ms

  1. Steps 1–2 same as HIT (DNS + TLS at PoP)
  2. Edge checks cache → object not found or TTL expired
  3. Edge forwards request to origin shield (mid-tier cache)
  4. Shield miss → forwards to origin; origin generates response
  5. Response cached at shield → forwarded to edge → edge caches it → user receives it
  6. Subsequent requests for same object are cache HITs at the PoP

Key Cache-Control Headers

The CDN respects Cache-Control headers from origin to determine cacheability and TTL:

  • Cache-Control: max-age=86400 — cache for 24 hours (both browser and CDN)
  • Cache-Control: s-maxage=3600 — CDN-specific TTL override (browser ignores this)
  • Cache-Control: no-store — CDN must not cache; every request hits origin
  • Cache-Control: private — CDN must not cache; response is user-specific
  • Surrogate-Key / Cache-Tag — CDN-specific headers enabling tag-based purging

TLS Termination at the Edge

The TLS handshake is one of the most latency-expensive operations in HTTPS — it takes 1–2 network round trips just to negotiate cipher suites and exchange certificates. By terminating TLS at the nearest PoP (~10ms away), the CDN ensures the handshake completes locally. The PoP-to-origin connection uses a pre-warmed, persistent TLS tunnel, so the user only pays the PoP-proximity handshake cost.

11 ⚖️ Trade-offs

✅ Benefit❌ Cost / Limitation
Dramatically reduces latency for globally distributed users (~15ms vs. ~200ms) Cache misses add a network hop — if CHR is low, CDN makes performance worse, not better
Reduces origin load — high CHR means origin handles only a fraction of total requests Additional infrastructure complexity: TTL tuning, invalidation strategy, CDN vendor lock-in risk
TLS termination at edge eliminates long-distance handshake latency (~150ms savings) Non-cacheable content (personalized, private, no-store) receives zero benefit and incurs CDN overhead
DDoS absorption at edge — malicious traffic is scrubbed before reaching origin CDN egress bandwidth costs significant money at scale; high-traffic sites pay per-GB for CDN delivery
Origin shield concentrates cache-fill load, preventing thundering herd at origin Origin shield adds one more network hop on a miss, slightly increasing cache-miss latency (~30ms)
Anycast provides automatic PoP failover via BGP reconvergence BGP reconvergence during PoP failure takes seconds–minutes — brief traffic disruption window

12 🎯 Interview Application

When an interviewer asks / says:

  • "How would you handle users in Asia accessing your US-hosted service?"
  • "Your static assets are slow for users in Europe — what do you do?"
  • "How do you reduce load on your origin servers at scale?"
  • "Walk me through the full request path from a user to your content"

What you say / do: Introduce CDN during the high-level design phase when you identify geographically distributed users or heavy static asset serving. Draw the PoP layer explicitly between users and origin. State your assumption about cache hit ratio and tie it to the content type. Always mention origin shield if scale is large enough that cache-miss fan-out is a concern.

"If we add CDN, we get sub-30ms latency for our static assets globally and reduce origin load by ~95%, but we pay for CDN bandwidth and need to manage cache invalidation carefully. For this system serving static JS/CSS and user-uploaded images, CDN is the right call — but I'd explicitly exclude our personalized feed API from CDN since it's Cache-Control: private and would get zero CHR benefit."

13 ⚠️ Misconceptions & Gotchas

❌Misconception: CDN always improves performance
✅Reality: CDN only improves performance on a cache HIT. For non-cacheable content (personalized responses, no-store or private headers), CDN adds a network hop and makes latency slightly worse. Always estimate CHR before adding CDN.
❌Misconception: CDN is only for static files (images, JS, CSS)
✅Reality: Modern CDNs accelerate dynamic content too — through TCP connection pre-warming (persistent tunnels to origin), edge compute (Cloudflare Workers run arbitrary logic at the PoP), and short-TTL caching. The correct framing is cacheable vs. non-cacheable, not static vs. dynamic.
❌Misconception: DNS geo-routing is as accurate as anycast
✅Reality: DNS geo-routing maps the DNS resolver's IP, not the user's IP, to a PoP. Corporate or ISP resolvers may be in a different city or country from the user, causing incorrect PoP selection. Anycast uses BGP at the packet level — it routes based on actual network topology and is far more accurate.
❌Misconception: You need to build your own CDN for very large scale
✅Reality: Even Netflix uses a CDN (Open Connect) — but that's exceptional and driven by ISP transit cost elimination at hundreds-of-Gbps scale. For virtually all systems at interview scale, a managed CDN (Cloudflare, CloudFront, Fastly) is correct. Building your own is a distraction unless you're designing "Netflix at Netflix scale."
