6.3 Cache Invalidation at the Edge
TTL, Versioned URLs & Active Purge

§0 🗺️ Topic Overview

Cache invalidation at the edge is how a globally distributed CDN serves fresh content without giving up the latency and origin-offload benefits that made you put a CDN in front of your origin. The core tension is freshness vs. propagation: a CDN has hundreds of independent edge caches spread across the planet, and there is no single "delete" button that clears them all instantly. You choose how each object becomes stale — passively by a TTL clock, by getting a brand-new URL, or by an active purge that fans out to every PoP.

  • 1. The three strategies and when to use each — TTL expiry, versioned URLs, and active purge. Match each to a content type: immutable assets → versioned URLs; volatile HTML/API → short TTL + purge; scheduled content → tuned TTL.
  • 2. Cache-Control semantics — max-age vs. s-maxage, stale-while-revalidate, stale-if-error, no-cache vs. no-store, immutable. These headers are the passive invalidation policy.
  • 3. Active purge mechanics and limits — purge-by-URL, purge-by-tag (surrogate keys), purge-everything. Know purge is not globally instant, is billed/rate-limited, and can stampede the origin.
  • 4. Revalidation with ETag / Last-Modified — conditional requests return a cheap 304 Not Modified instead of re-downloading the body when nothing changed.
  • 5. The browser-cache boundary — purging the CDN does nothing to copies already in end-user browsers. That's the whole reason versioned URLs exist.

§1 🎯 Goal of This Subtopic

After studying this, you should be able to design a complete edge-invalidation strategy for a given system — the Cache-Control policy per content type, when to use versioned URLs versus active purge, how to group related objects for bulk invalidation (cache tags / surrogate keys), and how to avoid overwhelming the origin when many objects are invalidated at once. You should be able to reason about the staleness window each approach creates and defend the trade-off out loud.

§5 🧠 Core Definition

Cache invalidation at the edge is the practice of controlling when and how a CDN's distributed edge caches stop serving a stale copy of an object and fetch or reveal a fresh one — via TTL-based expiry, versioned URLs, or active purge — so content stays acceptably fresh across all PoPs without sacrificing the cache-hit rate that makes the CDN valuable.

🔵 TTL passive

Edge serves a copy until its s-maxage clock runs out, then revalidates or re-fetches.

Simplest; staleness bounded by the TTL you chose.

🟢 Versioned URL rename

New content gets a new filename (app.a3f8b2.js), so the old object is never served again.

✅ Cache forever; also refreshes browser caches.

🟠 Active Purge push

API command evicts an object from every PoP before TTL — by URL, tag, or everything.

⚠️ Not globally instant; can stampede origin.

§6 📦 Core Concepts

TTL-Based Expiration (Passive Invalidation)

Every cached object carries a freshness lifetime, set primarily by Cache-Control (s-maxage for shared/CDN caches, max-age for browsers) or the legacy Expires header. While the object is "fresh," the edge serves it directly. Once the TTL elapses, the object becomes "stale" and the edge must revalidate it with the origin or re-fetch it before serving. This is the simplest strategy and needs no active signal — but the maximum staleness a user can experience equals the TTL you chose.

Versioned / Fingerprinted URLs (Cache Busting)

Instead of invalidating an object, you give the new version a different URL — typically by embedding a content hash in the filename (app.a3f8b2c9.js). Because the URL is new, no cache anywhere holds it, so it's fetched fresh; the old URL is simply never requested again and ages out. This lets you set an effectively infinite TTL (immutable, max-age=31536000) on static assets while still "updating" them instantly on deploy. It is the only technique that also refreshes copies sitting in end-user browsers.

Active Purge (Explicit Invalidation)

A control-plane command — via API or dashboard — that tells the CDN to evict an object before its TTL expires. Three granularities matter: purge by URL (one exact object), purge by tag / surrogate key (all objects sharing a label), and purge everything (flush the entire cache). Purge gives near-real-time freshness for content you cannot rename, but it is not globally instant, is often rate-limited and billed, and — at broad scope — can stampede the origin.

📊 Stale Object Revalidation — 304 vs 200

A stale-but-unchanged object costs a tiny 304 round-trip, not a full re-download

Revalidation with ETag / Last-Modified

When a stale object needs checking, the edge sends a conditional request using If-None-Match (with the object's ETag) or If-Modified-Since (with its Last-Modified date). If nothing changed, the origin replies 304 Not Modified with no body — the edge refreshes the object's TTL and keeps serving the copy it already has. This makes expiry cheap: a stale-but-unchanged object costs a tiny round-trip, not a full re-download.

Cache Tags / Surrogate Keys

A mechanism for grouping related cached objects so they can be invalidated together. The origin attaches tags to each response (e.g., Fastly's Surrogate-Key: product-123 category-shoes). When product 123 changes, a single purge of the product-123 tag invalidates every cached page and fragment that referenced it — the homepage, the category page, the product page — without you enumerating their URLs. This is how large content sites keep dynamic pages fresh without brittle URL lists.

§7 🔍 First Principles — Why Does This Exist?

A CDN exists to serve copies of your content close to users, which means the same object physically exists in hundreds of independent caches scattered across the globe. The moment your origin changes that object, every one of those copies is potentially wrong. Unlike a single database row you can update in place, there is no shared memory and no atomic transaction that spans every PoP — invalidation is inherently a distributed problem, and distributed systems cannot make a change appear everywhere simultaneously. Even a "fast" purge must propagate over the network to every edge node, and some receive the message milliseconds or seconds after others.

So the root problem is: you cannot have perfect freshness and maximum cache-hit rate at the same time across a planet-scale cache. Every design decision negotiates that tension. TTLs were the first answer — let each copy expire on its own clock, accepting bounded staleness for zero coordination. Versioned URLs were a clever sidestep: if you never reuse a URL, you never have to invalidate one, so you can cache forever. Active purge came last, for content you genuinely must update in place and cannot rename — accepting the cost, rate limits, and propagation delay of pushing a signal to every PoP.

🌍 Why Purge Isn't Instant — Fan-Out to Every PoP

A purge fans out PoP-by-PoP — briefly, different regions serve different versions

§8 🗺️ Mental Models

Model 1: Whiteboards in Every Branch Office

Your content is a notice written on a whiteboard in every one of your 300 branch offices (the PoPs). TTL is telling each office "erase this after 60 minutes." A versioned URL is posting a brand-new notice with a new title — nobody confuses it with the old one, which just gets ignored until wiped. Active purge is phoning every office to erase notice #123 right now — effective, but the calls take time and some offices pick up before others. Where it breaks down: real PoPs also revalidate (call HQ to check if the notice changed) rather than always erasing blindly.

Model 2: Pull vs. Push Invalidation

Frame the strategies by who initiates freshness. TTL and revalidation are pull: the edge lazily checks/expires on its own schedule; the origin does nothing special. Active purge is push: the origin actively broadcasts "drop this" to every edge. Versioned URLs are neither — they avoid the problem by making staleness impossible for a given URL. This maps directly onto cost and latency: pull is cheap and eventual; push is immediate but expensive and must fan out. Where it breaks down: stale-while-revalidate blends the two — the edge serves stale (pull) while refreshing in the background.

Model 3: The Rename Trick

The cleanest invalidation is the one you never have to do. Versioned URLs work on the principle that an object under a URL you never reuse can be treated as immutable — and immutable data is trivially cacheable forever, at every layer, including the browser. Whenever you're designing a purge pipeline for content that could instead be fingerprinted, ask: "Can I just rename it on change?" For build artifacts the answer is almost always yes. Where it breaks down: content with a stable, meaningful URL that users bookmark or that SEO depends on (like /products/123) cannot be renamed on every change — that's precisely the content that needs TTL + purge.

§9 ⚙️ How It Works — Mechanics

🚀 Versioned URL Deploy — Update Without Purge

Versioned URLs: a deploy changes the filename — instant update with zero purge calls

Passive Path (TTL + Revalidation)

Origin responds with a body plus Cache-Control: s-maxage=60 and an ETag. The edge marks it fresh for 60s and serves hits directly. After 60s the object is stale; the next request triggers a conditional GET with If-None-Match. Unchanged → 304 (headers only), refresh TTL, keep serving. Changed → 200 with new body and validators. stale-while-revalidate=30 lets the edge serve the stale copy immediately and revalidate in the background, so no user waits on the origin round-trip.

Active Purge Path

A content change triggers a purge — an API call with a URL or a Surrogate-Key. The control plane fans it out to every PoP (Fastly ~150ms; others seconds). Until a PoP receives it, that PoP may still serve the old copy. Soft purge marks the object stale (next request revalidates, can serve stale-if-error) — no hard cache gap. Hard purge deletes outright — the next request is a full miss. Tag purge invalidates every object bearing the key in one call.

Failure & Edge Cases

Thundering herd on broad purge: purge-everything (or a very hot object) makes many PoPs miss at once and fetch concurrently from origin. Mitigate with soft purge, request coalescing at the shield/tiered-cache layer, and origin request collapsing. Propagation race: during the seconds a purge propagates, some users hit updated PoPs and some don't — briefly inconsistent. Browser cache is out of reach: purge only affects the CDN; a browser holding max-age content won't re-fetch until its own TTL expires.

⚠️ Mass-purge thundering herd: "Purge everything" makes every PoP miss simultaneously, so N edges fire concurrent origin fetches for the hottest objects at the same instant — an origin that was comfortably shielded now takes a synchronized flood. Prefer targeted purge-by-tag; if a broad flush is unavoidable, use soft purge, pre-warm hot objects, or add a shield tier with request coalescing (SET NX-style single-flight).

What You Say / Do

This surfaces in the CDN/caching portion of a design — usually the read-path deep dive. Split by content type: "For static assets I'd fingerprint filenames and serve them immutable, max-age=1 year, so a deploy just changes the URL — instant update, no purge, and it even refreshes browser caches. For HTML and API responses I can't rename, I'd use a short s-maxage with stale-while-revalidate, tag each response with a surrogate key like product-123, and on a content change purge that tag — one call invalidates every page that shows the product. 💬 Trade-off statement (memorize this):

"Versioned URLs give me instant, purge-free updates and cache-forever hit rates, but only for content I can rename. For content with stable URLs, active tag-based purge gives near-real-time freshness at the cost of propagation delay and origin-stampede risk. For this system — fingerprinted assets plus event-driven HTML changes — I'd combine immutable versioned assets with short-TTL, surrogate-key-purged HTML, and never purge everything on the hot path."

§13 ⚠️ Common Misconceptions & Gotchas

❌
Misconception: Purging the CDN makes every user immediately see the new content.
✅
Reality: Purge only affects the CDN's edge caches, and even that isn't globally instant — it propagates PoP by PoP over milliseconds to seconds. Content already cached in end-user browsers is untouched and keeps serving until its own max-age expires. That browser-cache gap is exactly why versioned URLs exist for anything browsers cache aggressively.
❌
Misconception: A shorter TTL is always safer because content stays fresher.
✅
Reality: Shorter TTLs lower your cache-hit rate and push more traffic to the origin. Too low and the CDN barely helps — you get frequent revalidations and origin load spikes. The right TTL balances tolerable staleness against origin protection; for truly volatile content, pair a modest TTL with active purge rather than shrinking the TTL toward zero.
❌
Misconception: max-age and s-maxage are interchangeable.
✅
Reality: max-age sets the lifetime in private (browser) caches; s-maxage sets it in shared (CDN/proxy) caches and overrides max-age there. You routinely want them different — e.g., a long CDN TTL with active purge, but a short or zero browser TTL so users don't hold stale HTML you can't purge from their machines.
❌
Misconception: "Purge everything" is a safe, clean way to guarantee freshness.
✅
Reality: Flushing the whole cache makes every PoP miss simultaneously, so many edges fire concurrent origin fetches for the hottest objects at the same instant — a thundering herd that can overwhelm an origin that was comfortably shielded a moment earlier. Prefer targeted purge-by-tag; if you must flush broadly, use soft purge, pre-warming, or request coalescing at a shield tier.
